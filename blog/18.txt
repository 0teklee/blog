(예를 들면 useFragment훅을 이용한 data masking)</li></ul><p><br></p><p><br></p><p><strong>Relay의 단점</strong></p><p><br></p><ul><li>진입 장벽이 확실히 높다.</li><li>레퍼런스가 부족한 편이다.</li></ul><p><br></p><p>graphql과 마찬가지로 아직 많은 단점을 찾지는 못했다. 비슷한 기능을 공유하는 Apollo나 SWR을 graphql과 함께 사용해본다면 그 차이를 더 분명히 느낄 수 있을 것같다. 한가지 첨언하자면, apollo client나 swr보다 확실히 진입장벽이 있다는 점. 처음 적응할 때 새로운 프레임워크를 학습하는 정도의 난이도가 있었다.</p><p><br></p><p>학습에 큰 비용이 들 수 있으나, 그럼에도 relay를 사용하는 것은 엄청나다. 프론트 개발에서 undefined, null 에러가 지겹고, api 명세서를 일일히 봐가며 데이터의 타입 지정해야하는 것이 귀찮고, useState를 이용해 요청을 보내고, 응답을 받아 또다시 state로 ui를 업데이트 해야하는 게 불완전하게 느껴진다면 훌륭한 대안이 될 수 있다. 성능을 개선하는 것 뿐만 아니라,<strong>&nbsp;불필요한 코드를 상당 부분 줄여줄 수 있다</strong>.</p><p><br></p><p><br></p><h2>Relay의 기본 요소 &amp; Hooks</h2><p><br></p><ul><li><strong>Relay Compiler</strong></li></ul><p><br></p><blockquote>The&nbsp;<a href="https://relay.dev/docs/guides/compiler/" rel="noopener noreferrer" target="_blank" style="color: var(--ifm-link-color);">Relay Compiler</a>&nbsp;will analyze any&nbsp;graphql&nbsp;literals inside your Javascript code, and produce a set of artifacts that are used by Relay at runtime, when the application is running on the browser. - Relay docs</blockquote><p><br></p><p>릴레이를 통해 graphql을 사용하기 위해선 런타임 이전에 릴레이 컴파일을 해야한다. 그 결과로 query에 대응하는 artifact 파일들이 생성된다. "<em>릴레이 컴파일러는 js 코드 내의 모든 graphql 리터럴을 분석하고, 이를 바탕으로 브라우저가 실행될 때 런타임에서 사용할&nbsp;</em><strong><em>artifact&nbsp;</em></strong><em>세트를 생성한다</em>". (일반적으로 __generated__ 폴더에 저장된다.) artifact는 릴레이 컴파일러가 실행될 때마다 생성되어 프로젝트에서 변경된 gql을 반영한다. graphql 태그 자체가 런타임에서 실행되는 것이 아니다. 컴파일러가 생성한 artifact를 기반으로 relay-runtime이 실행되어 graphql이 작동할 수 있다.</p><p><br></p><p>개인적으로 컴파일러를 실행하여 쿼리 내부&nbsp;<strong>타입을 미리 검증하고</strong>, 컴파일 이후 생성된 artifact에서 데이터의<strong>&nbsp;타입을 바로 가져올 수 있는 점이 상당히 혁신적</strong>으로 다가왔다. TS를 사용한다면 types.ts에 무수히 많은 인터페이스와 타입 등을 작성했어야 했을 것이다. 반복적일 뿐만 아니라, ts가 던지는 컴파일 에러가 무시되고 런타임이 실행되는 경우도 종종 있기에 이 작업이 무의미해질 때도 있다.</p><p><br></p><p><br></p><ul><li><strong>Relay Store</strong></li></ul><p><br></p><blockquote><span style="color: rgb(28, 30, 33);">The Relay Store can be used to programmatically update client-side data inside&nbsp;</span><a href="https://relay.dev/docs/guided-tour/updating-data/graphql-mutations/" rel="noopener noreferrer" target="_blank" style="color: inherit;">updater</a><a href="https://relay.dev/docs/guided-tour/updating-data/graphql-mutations/" rel="noopener noreferrer" target="_blank" style="color: var(--ifm-link-color);">&nbsp;functions</a><span style="color: rgb(28, 30, 33);">. The following is a reference of the Relay Store interface.</span></blockquote><p><br></p><p>릴레이 스토어는 일반적인 상태관리툴의 저장소 store와 유사하다. 일반적인 상태관리 툴에서 상태를 변경하는 dispatch 기능이 updater 함수를 통해 구현된다. 다만 중요한 차이점은 릴레이 스토어는 모든 요청과 응답에 대한 데이터가 저장된다는 점이다. 또한 데이터의 상태를 변경하기 위해 복잡한 dispatch를 요구하지 않는다. 릴레이에서 모든 쿼리는 스토어를 거치기 때문이다. 이를 통해 불필요한&nbsp;<a href="https://limelightkr.co.kr/round-trip-time%EB%9E%80-%EB%AC%B4%EC%97%87%EC%9E%85%EB%8B%88%EA%B9%8C/" rel="noopener noreferrer" target="_blank" style="color: inherit;">round trip</a>을 방지할 수 있다.</p><p><br></p><p>덕분에 다양한 성능 최적화를 이룰 수 있다. 예를 들어 optimistic response와 usePreloadedQuery 같은 다양한 훅이 이 스토어를 기반으로 작동한다. optimistic response을 활용하면, 동일한 쿼리로 요청을 보내 스토어에 있는 값과 동일한 응답을 받을 것이라 예상된다면 아예 요청을 보내지 않는다. (<a href="https://www.apollographql.com/docs/react/v2/performance/optimistic-ui/" rel="noopener noreferrer" target="_blank" style="color: inherit;">Optimistic UI</a>) 혹은 Suspense을 함께 사용하여&nbsp;<a href="https://17.reactjs.org/docs/concurrent-mode-suspense.html" rel="noopener noreferrer" target="_blank" style="color: inherit;">render as you fetch</a>&nbsp;방식으로 페이지를 로드할 수 있다. 즉 렌더링과 데이터 페칭을 동시에 구현할 수 있다. 뿐만 아니라 스토어에는 다양한 정보가 저장된다. 이를테면 query에 입력된 argument를 불러오거나 할 수 있다. 이처럼 스토어는 컴파일러와 더불어 릴레이의 강력한 기능의 기반이 된다고 할 수 있다.</p><p><br></p><p><br></p><ul><li><strong>Relay Environment</strong></li></ul><p><br></p><blockquote><span style="color: rgb(28, 30, 33);">A Relay Environment instance to execute the request on. (예를 들면 useFragment훅을 이용한 data masking)</li></ul><p><br></p><p><br></p><p><strong>Relay의 단점</strong></p><p><br></p><ul><li>진입 장벽이 확실히 높다.</li><li>레퍼런스가 부족한 편이다.</li></ul><p><br></p><p>graphql과 마찬가지로 아직 많은 단점을 찾지는 못했다. 비슷한 기능을 공유하는 Apollo나 SWR을 graphql과 함께 사용해본다면 그 차이를 더 분명히 느낄 수 있을 것같다. 한가지 첨언하자면, apollo client나 swr보다 확실히 진입장벽이 있다는 점. 처음 적응할 때 새로운 프레임워크를 학습하는 정도의 난이도가 있었다.</p><p><br></p><p>학습에 큰 비용이 들 수 있으나, 그럼에도 relay를 사용하는 것은 엄청나다. 프론트 개발에서 undefined, null 에러가 지겹고, api 명세서를 일일히 봐가며 데이터의 타입 지정해야하는 것이 귀찮고, useState를 이용해 요청을 보내고, 응답을 받아 또다시 state로 ui를 업데이트 해야하는 게 불완전하게 느껴진다면 훌륭한 대안이 될 수 있다. 성능을 개선하는 것 뿐만 아니라,<strong> 불필요한 코드를 상당 부분 줄여줄 수 있다</strong>.</p><p><br></p><p><br></p><h2 id="relay-hooks">Relay의 기본 요소 &#x26; Hooks</h2><p><br></p><ul><li><strong>Relay Compiler</strong></li></ul><p><br></p><blockquote>The <a href="https://relay.dev/docs/guides/compiler/" rel="noopener noreferrer" style="color: var(--ifm-link-color);">Relay Compiler</a> will analyze any graphql literals inside your Javascript code, and produce a set of artifacts that are used by Relay at runtime, when the application is running on the browser. - Relay docs</blockquote><p><br></p><p>릴레이를 통해 graphql을 사용하기 위해선 런타임 이전에 릴레이 컴파일을 해야한다. 그 결과로 query에 대응하는 artifact 파일들이 생성된다. "<em>릴레이 컴파일러는 js 코드 내의 모든 graphql 리터럴을 분석하고, 이를 바탕으로 브라우저가 실행될 때 런타임에서 사용할 </em><strong><em>artifact </em></strong><em>세트를 생성한다</em>". (일반적으로 __generated__ 폴더에 저장된다.) artifact는 릴레이 컴파일러가 실행될 때마다 생성되어 프로젝트에서 변경된 gql을 반영한다. graphql 태그 자체가 런타임에서 실행되는 것이 아니다. 컴파일러가 생성한 artifact를 기반으로 relay-runtime이 실행되어 graphql이 작동할 수 있다.</p><p><br></p><p>개인적으로 컴파일러를 실행하여 쿼리 내부 <strong>타입을 미리 검증하고</strong>, 컴파일 이후 생성된 artifact에서 데이터의<strong> 타입을 바로 가져올 수 있는 점이 상당히 혁신적</strong>으로 다가왔다. TS를 사용한다면 types.ts에 무수히 많은 인터페이스와 타입 등을 작성했어야 했을 것이다. 반복적일 뿐만 아니라, ts가 던지는 컴파일 에러가 무시되고 런타임이 실행되는 경우도 종종 있기에 이 작업이 무의미해질 때도 있다.</p><p><br></p><p><br></p><ul><li><strong>Relay Store</strong></li></ul><p><br></p><blockquote><span style="color: rgb(28, 30, 33);">The Relay Store can be used to programmatically update client-side data inside </span><a href="https://relay.dev/docs/guided-tour/updating-data/graphql-mutations/" rel="noopener noreferrer" style="color: inherit;">updater</a><a href="https://relay.dev/docs/guided-tour/updating-data/graphql-mutations/" rel="noopener noreferrer" style="color: var(--ifm-link-color);"> functions</a><span style="color: rgb(28, 30, 33);">. The following is a reference of the Relay Store interface.</span></blockquote><p><br></p><p>릴레이 스토어는 일반적인 상태관리툴의 저장소 store와 유사하다. 일반적인 상태관리 툴에서 상태를 변경하는 dispatch 기능이 updater 함수를 통해 구현된다. 다만 중요한 차이점은 릴레이 스토어는 모든 요청과 응답에 대한 데이터가 저장된다는 점이다. 또한 데이터의 상태를 변경하기 위해 복잡한 dispatch를 요구하지 않는다. 릴레이에서 모든 쿼리는 스토어를 거치기 때문이다. 이를 통해 불필요한 <a href="https://limelightkr.co.kr/round-trip-time%EB%9E%80-%EB%AC%B4%EC%97%87%EC%9E%85%EB%8B%88%EA%B9%8C/" rel="noopener noreferrer" style="color: inherit;">round trip</a>을 방지할 수 있다.</p><p><br></p><p>덕분에 다양한 성능 최적화를 이룰 수 있다. 예를 들어 optimistic response와 usePreloadedQuery 같은 다양한 훅이 이 스토어를 기반으로 작동한다. optimistic response을 활용하면, 동일한 쿼리로 요청을 보내 스토어에 있는 값과 동일한 응답을 받을 것이라 예상된다면 아예 요청을 보내지 않는다. (<a href="https://www.apollographql.com/docs/react/v2/performance/optimistic-ui/" rel="noopener noreferrer" style="color: inherit;">Optimistic UI</a>) 혹은 Suspense을 함께 사용하여 <a href="https://17.reactjs.org/docs/concurrent-mode-suspense.html" rel="noopener noreferrer" style="color: inherit;">render as you fetch</a> 방식으로 페이지를 로드할 수 있다. 즉 렌더링과 데이터 페칭을 동시에 구현할 수 있다. 뿐만 아니라 스토어에는 다양한 정보가 저장된다. 이를테면 query에 입력된 argument를 불러오거나 할 수 있다. 이처럼 스토어는 컴파일러와 더불어 릴레이의 강력한 기능의 기반이 된다고 할 수 있다.</p><p><br></p><p><br></p><ul><li><strong>Relay Environment</strong></li></ul><p><br></p><blockquote><span style="color: rgb(28, 30, 33);">A Relay Environment instance to execute the request on.
