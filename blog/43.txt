정적 파일을 제공하기 때문에 로드 시간 면에서 가장 빠름</li></ul><p><br></p><p><strong>ISR </strong>(증분 정적 재생성 getStaticProps)<strong>:</strong></p><p><br></p><ul><li><strong>범위</strong>: SSG와 유사하지만, 페이지를 설정된 간격이나 요청에 따라 다시 컴포넌트:</strong></p><p><br></p><ul><li><strong>범위</strong>: 페이지 내 특정 컴포넌트들이 서버에서 렌더링됨</li><li><strong>사용 용도</strong>: 인터랙티브하지 않거나 계산이 많이 필요한 컴포넌트에 적합. 클라이언트 측 상호작용이 필요하지 않을 때</li><li><strong>성능</strong>: 클라이언트로 보내는 자바스크립트의 양을 줄임. 서버에서 렌더링을 처리함으로써 전체 성능 향상</li><li>어떤 부분이 클라이언트 측의 동적인 상호작용과 렌더링을 필요로 하고, 어떤 부분을 효율적으로 서버에서 렌더링할 수 있는지 전략적으로 결정할 수 있게됨.</li><li>이 렌더링 방식의 세분화는 클라이언트 측의 부하를 줄이고 서버 자원을 더 효과적으로 활용할 수 있게 하여 성능을 최적화하는 데 도움이 됨</li></ul><p><br></p><h3><strong>서</strong>버 컴포넌트 구성 패턴</h3><p><br></p><p><img src="http://res.cloudinary.com/dolziw8fv/image/upload/v1704720835/micrqu1aa6kzxuckzkas.png"></p><p><br></p><p>Next.js 공식 문서에서 어느 상황에서 서버/클라이언트 컴포넌트를 사용해야하는지 잘 설명하고 있다.&nbsp;서버/클라이언트 구분이 낯설고 불필요하게 느껴질 noreferrer" target="_blank">서버와 클라이언트 구성 패턴</a>에는 이러한 새로운 제약을 활용하는 방법을 설명하고 있다. 이 중 앞서 서버 컴포넌트 사용 시 "클라이언트로 보내는 자바스크립트의 양을 줄인다"는 내용이 있었는데, 다음과 같은 간단한 패턴을 통해 이룰 수 있다. <a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" rel="noopener noreferrer" target="_blank"><strong>클라이언트 컴포넌트를 렌더링 트리 아래로 옮기기</strong></a><a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" rel="noopener noreferrer" target="_blank" style="color: rgb(45, 16, 176);"><strong>M</strong></a><a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" 컴포넌트</strong>: 페이지 내 <u>특정 부분(컴포넌트)</u>들이 서버 또는 클라이언트에서 렌더링되는지 결정할 수 있음</p><p><br></p><p><strong>SSR </strong>(getServerSideProps)<strong>:</strong></p><p><br></p><ul><li><strong>범위</strong>: 각 요청마다 생성할 수 있음</li><li><strong>사용 용도</strong>: 주기적으로 변경되는 컨텐츠에 유용하며, 정적과 동적 컨텐츠의 장점을 살릴 수 있음</li><li><strong>성능</strong>: SSG의 속도 장점과 SSR의 서버 데이터의 최신 상태라는 장점을 결합</li></ul><p><br></p><p><strong>React 서버 컴포넌트:</strong></p><p><br></p><ul><li><strong>범위</strong>: 페이지 내 특정 컴포넌트들이 서버에서 렌더링됨</li><li><strong>사용 용도</strong>: 인터랙티브하지 않거나 계산이 많이 필요한 컴포넌트에 적합. 클라이언트 측 상호작용이 필요하지 않을 때</li><li><strong>성능</strong>: 클라이언트로 보내는 자바스크립트의 양을 줄임. 서버에서 렌더링을 처리함으로써 전체 성능 향상</li><li>어떤 부분이 클라이언트 측의 동적인 상호작용과 렌더링을 필요로 하고, 어떤 부분을 효율적으로 서버에서 렌더링할 수 있는지 전략적으로 결정할 수 있게됨.</li><li>이 렌더링 방식의 세분화는 있지만, 개인적으로 컴포넌트에 대한 구분이 명확해지면서 설계 패턴 또한 한층 더 분명해지는 느낌을 받았다. 이는 프론트 코드에서 복잡성이 증가하게되는 원인이 주로 서버 데이터와 연관되기 때문이다. api의 호출 시점, 데이터의 형태, 분기 처리 등 서버 데이터 상태와 UI 상태를 동시에 효과적으로 관리해야한다. 신경써야할 것들이 많아질 수록 코드는 복잡해지며, 컴포넌트 자체도 UI만 표시하는 역할에서 점차 많은 역할을 떠안게 된다. 서버 컴포넌트는 코드가 구조적으로 복잡해지는 상황을 통제하기 위한 효율적인 대안으로 보인다.</p><p><br></p><p>서버 컴포넌트를 통해 서버 데이터를 호출 및 가공하여 데이터를 상속하고, 이를 상속받은 클라이언트 컴포넌트는 UI의 상태와 이벤트 로직을 관리한다. UI의 상태를 관리하는 useState, useEffect와 같은 훅들과 이벤트 핸들러처럼 사용자의 인터랙션을 전제로 하는 컴포넌트는 모두 클라이언트 컴포넌트로 볼 수 있다. 이러한 구분을 통해 더 명확한 관심사의 분리를 이룰 수 있다. </p><p><br></p><p>기본적으로 서버 컴포넌트 자체에 함수형 컴포넌트에 async를 붙인다는 큰 차이가 있다. 이를 감안하면 서버 컴포넌트와 클라이언트 컴포넌트는 다음과 단순한 규칙을 따르게 된다.</p><p><br></p><ol><li>서버 컴포넌트는 최상단에서 사용된다</li><li>클라이언트 컴포넌트는 서버 컴포넌트를 import하여 return에서 호출할 수 없다.</li><li>클라이언트 컴포넌트는 {props.children}으로 서버 컴포넌트를 감쌀 수 있다.</li></ol><p><br></p><p>Next.js 공식문서의 <a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" rel="noopener noreferrer">서버와 클라이언트 구성 패턴</a>에는 이러한 새로운 제약을 활용하는 방법을 설명하고 있다. 이 중 앞서 서버 컴포넌트 사용 시 "클라이언트로 보내는 자바스크립트의 양을 줄인다"는 내용이 있었는데, 다음과 같은 간단한 패턴을 통해 이룰 수 있다. <a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" rel="noopener noreferrer"><strong>클라이언트 컴포넌트를 렌더링 트리 아래로 옮기기</strong></a><a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" rel="noopener noreferrer" style="color: rgb(45, 16, 176);"><strong>M</strong></a><a href="https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree" rel="noopener noreferrer"><strong>oving Client Components Down the Tree</strong></a>. 이는 앞서 얘기한 관심사 분리의 이점과도 일맥상통하는 부분이 있다.</p><p><br></p><p>링크의 내용을 요약하자면, 사용자 상호작용이 있는 컴포넌트와 그것이 불필요한 컴포넌트를 분리하는 간단한 예시가 나온다. 즉 이벤트를 관리하는 UI 컴포넌트 (클라이언트)와 서버 컴포넌트를 분리하는 패턴이다. 예시에는 헤더에서 서버 데이터를 필요로 하지 않는 로고와 사용자가 입력 가능한 검색바를 클라이언트 컴포넌트를 이용해 분리한다. 이벤트가 일어나는 검색바는 클라이언트 컴포넌트로 변경한다. 이런 간단한 변경을 통해, 자바스크립트 번들 사이즈 줄이는 것이 가능해진다. 이 예시에서 클라이언트 컴포넌트인 검색바SearchBar를 제외한 나머지는 별도의 자바스크립트 없이 미리 렌더링되어 브라우저로 보내진다.</p><p>사실 예시에서 짚은 것은 헤더, 푸더의 위치를 잡는 layout 컴포넌트에 대한 이야기였는데, 동적인 ui 이벤트가 발생하지 않고 컴포넌트의 위치와 구성을 잡는 이같은 컴포넌트는 서버 컴포넌트를 사용해야한다. 그리고 App router를 사용한다면 기본적으로 layout.tsx를 활용할 수 있다.
