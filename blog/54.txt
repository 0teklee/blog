(<a href="https://www.teklog.site/blog/22" rel="noopener noreferrer" target="_blank">이전 글 참조</a>)</p><p><br></p><p>스택의 기본적인 방법으로 push와 pop을 통해 아이템을 스택 상단에 추가하고 제거한다. 자바스크립트 배열 메서드를 떠올리면 쉽다.</p><p><br></p><p><strong><em>React의 원래 reconciler는 오래된 가상 트리와 새로운 가상 트리를 비교하고 DOM을 업데이트하는 데 사용되는 스택 기반 알고리즘</em></strong><em>이었습니다.&nbsp;</em><strong><em>stack reconciler는 간단한 경우에는 잘 작동했지만, 애플리케이션이 크기와 복잡성에서 성장함에 따라 여러 도전적인 상황을 보여줬습니다</em></strong></p><p><br></p><p><strong>도전적인 상황의 예시</strong></p><p><br></p><ul><li>중요하지 않지만 연산적으로 비용이 많이 드는 컴포넌트(ExpensiveComponent)가 CPU를 소비하며 렌더링</li><li>사용자가 입력 요소에 타이핑</li><li>입력이 유효하면 버튼이 활성화</li><li>Form 컴포넌트가 상태를 가지므로, 리렌더링</li></ul><pre class="ql-syntax" spellcheck="false">import React, { useReducer } from "react";

const initialState = { text: "", isValid: false };

function Form() {
  const [state, dispatch] = useReducer(reducer, initialState);

  const handleChange = (e) =&gt; {
    dispatch({ type: "handleInput", payload: e.target.value });
  };

  return (
    &lt;div&gt;
      &lt;ExpensiveComponent /&gt;
      &lt;input value={state.text} onChange={handleChange} /&gt;
      &lt;Button disabled={!state.isValid}&gt;Submit&lt;/Button&gt;
    &lt;/div&gt;
  );
}

function reducer(state, action) {
  switch (action.type) {
    case "handleInput":
      return {
        text: action.payload,
        isValid: action.payload.length &gt; 0,
      };
    default:
      throw new Error();
  }
}

</pre><p>스택 조정자reconciler는 작업을 일시 중지하거나 연기할 수 없이 스택의 상단부터 순차적으로 업데이트를 렌더링할 것이다. 여기서 다음과 같은 문제가 생긴다.</p><p><br></p><ul><li>연산적으로 비용이 많이 드는 컴포넌트가 텍스트 입력 렌더링을 차단</li><li>사용자 입력이 화면에 지연되어 렌더링</li><li>텍스트 필드가 반응하지 않는 것처럼 보여 사용성 저하</li></ul><p><br></p><p><strong>직관적인 해결법은 렌더링의 순서를 중요도에 따라 바꾸는 것이다</strong>. 중요하지 않지만 비용이 많이 드는 컴포넌트 렌더링은 나중에, 사용자 입력을 더 먼저 우선순위를 두고 먼저 렌더링한다. 즉 스택의 순서대로가 아닌 우선순위 기반으로 가상 DOM을 업데이트하여 해결할 수 있을 것이다. 그러기 위해서 리액트는 사용자 입력 같은 특정 유형의 렌더링 작업을 다른 렌더링보다 우선순위를 매길 수 있어야한다.</p><p><br></p><p>이같은 문제 인식과 해결법을 통해 기존 스택 조정자reconciler의 한계와 렌더링 우선순위 부여의 필요성을 이해할 수 있다.</p><p><br></p><p><strong>스택 조정자의 한계</strong></p><ul><li>우선순위에 따라 가상 DOM을 업데이트 하지 않음</li><li>업데이트를 중단하거나 취소할 수 없음</li><li>느린 업데이트, 불필요한 렌더링이 성능과 사용성 저하로 이어짐</li></ul><p><br></p><p>스택 조정자reconciler는 업데이트를&nbsp;<strong>우선순위에 따라 처리하지 않았다</strong>. 렌더링에 우선순위가 없기 때문에, 이는&nbsp;<strong>중요하지 않은 업데이트가 더 중요한 업데이트를 차단할 수 있었다</strong>.</p><p><br></p><p>스택 조정자reconciler의 또 다른 한계는&nbsp;<strong>업데이트를 중단하거나 취소할 수 없다는 것</strong>이다. 이는 스택 reconciler가 업데이트&nbsp;<strong>우선순위를 인식하더라도</strong>, 고 우선순위 업데이트가 예약되었을 때 낮은 우선순위의 작업을 취소하거나 중단할 수 없었다. 이는 곧 불필요한 업데이트가 이루어진다는 뜻이며, 가상 트리와 DOM에서&nbsp;<strong>불필요한 작업이 수행되어 애플리케이션의 성능에 부정적인 영향</strong>을 줄 수 있다.</p><p><br></p><p>업데이트 우선 순위의 예시들</p><ul><li>텍스트 입력은 마우스 호버 시에만 렌더링되는 툴팁 보다 높은 우선 순위를 갖는다</li><li>버튼 클릭은 notification 알림의 표시 보다 높은 우선순위를 갖는다.</li><li>input 입력이나 버튼 클릭은 즉각적인 반응을 요구하는 의도적인 행동이기 때문에 우선순위가 높다.</li></ul><p><br></p><p><strong>결론</strong></p><p>이전의 조정 방식은 스택 순서로 일관되게 업데이트를 처리하여 성능 저하, 사용성 저하로 이어졌다. 효율적인 순서로 가상 DOM을 업데이트할 필요성이 생겼고, 파이버 조정자Fiber reconciler가 개발되었다.</p><p><br></p><p><br></p><h2>파이버 조정자 Fiber Reconciler</h2><p><br></p><p><br></p><p><br></p><h3><strong>파이버 Fiber란?</strong></h3><ul><li>조정자 reconciler의 작업 단위</li><li>파이버 조정자를 통해 </pre><h5>completeWork</h5><p><br></p><p><span style="background-color: rgba(0, 0, 0, 0.05);">completeWork</span>&nbsp;함수는 새로운 실제 DOM 트리를 구성한다. 이 트리는 브라우저 밖에서 DOM과 분리되어 구성된다. 브라우저인 경우, 이는&nbsp;<span style="background-color: rgba(0, 0, 0, 0.05);">document.createElement</span>&nbsp;또는&nbsp;<span style="background-color: rgba(0, 0, 0, 0.05);">newElement.appendChild</span>와 같은 작업을 의미한다.&nbsp;<strong>하지만 이 DOM 트리는 아직 브라우저 내 document에 연결되지 않았다</strong>는 점이 중요하다.&nbsp;<strong>React는 화면 밖에서 UI의 다음 버전을 생성한다</strong>. Placement 효과</h5><p><br></p><p>이 효과는 새로운 컴포넌트가 DOM에 추가될 때 발생한다. 예를 들어, form에 새 버튼이 추가되면, 그 버튼을 DOM에 추가하기 위한 배치 효과가 발생한다.</p><p><br></p><h5>업데이트 효과</h5><p><br></p><p>컴포넌트가 새로운 props나 상태로 업데이트될 때 발생한다. 0.05);">commitRoot</span>는&nbsp;<strong>FiberRootNode의 포인터를 현재 트리에서 작업 진행 중인 트리로 전환하여, 작업 진행 중인 트리를 새로운 현재 트리로 만든다.</strong></li><li>업데이트 이후: 이때부터, 모든 업데이트는&nbsp;<strong>새로운 현재 트리를 기반으로 한다.</strong>&nbsp;이 과정은&nbsp;<strong>애플리케이션이 일관된 상태를 유지하고, 업데이트가 올바르고 효율적으로 적용되도록 보장</strong>한다.</li></ul><p><br></p><p>이것이 조정Reconcile이 작동하는
