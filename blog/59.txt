게 나을까?</p></li><li><p>각 조건을 키로 사용할 때 케이스를 비교한다.</p></li><li><p>숫자(점수)를 키로 사용: 값이 문자 배열이 된다. 문자 비교는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">{ [key: string // 선택된 쿼리 문자] : number[] // 지원자들의 점수}</code> 키와 값의 타입을 결정했다. 2번째 탐색에서 숫자가 유리하기 때문이다.</p></li><li><p>문자열 키의 조건은 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code> 문자 일부가 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code> 문자를 포함할 때이다.</p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 문자만으로 키를 찾을 수 있도록 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">hash(x)</code>의 적절한 방식을 고려한다.</p></li><li><p>이 때 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">hash(query[i])</code>를 사용해 바로 값을 얻을 수 있도록 해시를 설계한다.</p></li><li><p>바로 떠오르지 않는다. 전체 과정을 추론하는 단계이니,<code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">hash(query[i])</code> 키를 사용해 바로 값인 숫자(점수) 배열을 가져왔다고 가정한다.</p></li><li><p>해시로부터 점수 배열을 가져왔다. 어떻게 2단계 탐색을 할까?</p></li><li><p>점수 배열에는 x 이상만 정렬하면 된다. 이 때 O(n)이 되지 않게 filter()같은 메서드는 사용하지 않는다.</p></li><li><p>특정 숫자 이상의 범위만 알면 된다. 두 개의 포인터를 움직이는 탐색으로 O(n)을 피할 수 있다.</p></li><li><p>그러기 위해 점수 배열이 미리 sort가 이루어지면 더 편하겠다.</p></li><li><p>이 탐색이 끝나면, 해시에서 가져온 배열에서 x 이상인 요소 갯수를 알 수 있다. 배열의 갯수는 곧 길이이므로 right - left + 1로 파악 가능하다.</p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code> 사이클을 마지막에는 21에서 얻은 길이를 반환할 배열에 push한다.</p></li><li><p>모든 순회가 끝나면 저장된 답변을 반환한다.</p></li></ol><p></p><h4 class="mt-5 mb-3">적절한 해시 구조 찾기</h4><p>질문 :  16. 어떻게 문자 키<code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">hash(query[i])</code>로 바로 값을 얻을 수 있는 해시를 만들까?</p><p>"frontend" -&gt; "frontend"를 포함하는 모든 요소가 "front"의 값 배열에 들어가 있다<br>"java senior" -&gt; "java senior"를 포함하는 모든 요소가 "java senior" 키-값 배열에 들어있다</p><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 문자열 키는 1~4개의 필드가 될 수 있다. 키가 "frontend"인 것만으로 "frontend"를 포함하는 모든 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info</code> 요소가 값-배열에 담겨 한번에 가져올 수 있도록 해시가 구성되어야 한다. 그럼 해시에 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info</code>를 저장할 때 키가 한번에 여러 요소를 가져올 수 있도록 설계한다.</p><p></p><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info</code>에서 문자 필드 4개의 경우의 수는 2^4이다. 해시에 16 X info.length 만큼 키를 만들면 되지 않을까? <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info</code> 전체에 적용하면 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">O(n X 2^4) =&gt; O(n)</code>이므로, <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info</code>에 있는 모든 키의 조합을 저장하는 것이 가능하다. 해시 값에는 "frontend"인 모든 요소, "java frontend"인 모든 요소를 배열에 push한다. 중복이 생겨 공간 효율이 떨어지는 것 같지만, 쿼리가 선택적nullable이기 때문에 필요하다.</p><p>이로써 첫번째 조건의 구현 목표가 뚜렷해졌다. 이를 목표하여 모든 키의 조합을 키로 만드는 방법을 찾는다.</p><p></p><p><em>참고</em></p><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info</code>에서 문자 필드 4개의 경우의 수가 2^4인 이유</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>4가지 문자 필드의 모든 조합을 있거나("java")/ 없거나( "-")로 구분한다.</p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>는 해시에 저장된 키만 O(1)로 가져온다.<br>이게 문제의 함정으로 보인다. 언어 3개, 직군 2개, 경력 2개, 소울푸드 2개 - 필드가 가능한 숫자와 상관없이 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>의 문자 필드는 not null이기에, 저장 단계에서 필터링이 가능하게 만든다. <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 문자 필드는 "-"이거나 문자이다.</p></li></ol><h3 class="mt-5 mb-3">구현하기</h3><pre class="ql-syntax hljs rounded-md p-4 my-2 text-sm"><code class="language-js">function solution(info, query) {
    // 0. Map을 사용한 탐색 저장소.
    let map = new Map();

    // 1. 저장 단계 : map에 저장하기 위한 함수. 
    // info[i]의 문자열 키로 가능한 모든 조합 등록 + 해당 키-값인 배열에 점수 추가.
    // dfs가 사용되었다.
    function updateMap (words, score, start, key){
        if(start === words.length) { // key에 4개 단어가 완성됐을 때 키, 점수를 추가한다
            if(!map.has(key)) {
                map.set(key, [])
            }
            map.get(key).push(score)
            return // 재귀 함수의 중단 조건
        }
        // dfs로 가능한 모든 조합을 만든다. 
        updateMap(words, score, start + 1, key + words[start]) // 첫 호출 시 첫 키가 'words[0]'
        updateMap(words, score, start + 1, key + "-") // 첫 호출 시 처음 키는가 '-'
    }
    
    // 2. info를 map에 실제로 저장
    for(let 지원자 of info) {
        let words = 지원자.split(" ")
        let score = Number(words.pop()); // 문자열로 저장된 숫자이니 변환한다.
        updateMap(words, score, 0, '') // updateMap의 첫 호출은 빈 문자
    }
    // 3. map에 저장된 모든 키의 점수를 정렬한다.
    for(let [_, scores] of map){
        scores.sort((a,b) =&gt; a - b)
    }
    
    // 4. query의 O(n) 순회를 시작한다.
    let result = [];
    
    for(let q of query) {
        let qArr = q.replace(/ and /g, " ").split(" ");
        let qScore = Number(qArr.pop()) // X 이상인 조건을 숫자로 바꾼다.
        let qKey = qArr.join("") // map에서 가져오기 위한 조합의 키
        
        // 4-1 map에 없을 시 0을 추가하고 넘어간다.
        if(!map.has(qKey)){ 
            result.push(0) 
            continue
        } 
        // 4-2 키를 찾을 시 두번째 탐색, 점수 별로 정렬한다.
        let infoScores = map.get(qKey)
        // 이진 탐색으로 qScore 이상인 점수만 남긴다.
        let left = 0, right = infoScores.length
        //  right를 qScore와 가장 근접한 곳으로 옮기고, left로 점수 이하를 뺀다.
        while(left &lt; right) {
              let mid = Math.floor((left+right) / 2)
            if(infoScores[mid] &gt;= qScore) {
               right = mid
            } else {
                left = mid + 1
            }   
        }
        result.push(infoScores.length - left)
    }
    
    return result;
}</code></pre><p>15분의 제한 시간을 두고 풀었다.<br>풀기 전 과정은</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p>손으로 코드를 적어가며 원리를 되새겼다.</p></li><li><p>처음 나의 추론 과정과 정답 추론 과정을 읽었다.</p></li><li><p>이 과정에서 코드를 암기하려 하지 않고, <strong>추론 과정과 원리 이해</strong>에 집중했다.</p></li><li><p>3번까지 마친 후, 30~1시간 시간을 두고 처음부터 문제를 풀었다.</p></li></ol><h3 class="mt-5 mb-3">첫 번째 조건 - 해시 구조</h3><p>핵심: 적절한 데이터 형태를 위해 도구를 사용하기</p><p>여기서 배운 사고 과정은 이렇다.</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p>어떻게 해시를 설계할까<br>오답 추론에서는 이 질문에서 한번 더 구체화가 이루어지지 않았다. 단지 '키로 필터링하기'에만 집중했다.</p></li><li><p>문자열 키를 사용해 바로 숫자 배열이라는 값을 얻을 수 있는 해시 구조는 무엇일까<br>정답 추론 과정에서 여기까지는 질문이 구체화되었다. 키가 문자가 되고, 값이 숫자배열이어야 한다-는 필요성을 인지했다.</p></li><li><p>'무슨 알고리즘을 쓰면 키의 모든 조합을 맵에 저장할 수 있을까?'<br>마침내 필요한 질문까지 도달한 것 같다. 문자열 조합이라면 그래프 탐색이 적절하다.</p></li></ol><p>이 문제에선 해시 구조를 위해 DFS 알고리즘을 사용한다. 처음 정답 코드를 봤을 때, 그저 코드를 이해한 정도였다. 답은 알아도, 답에 이르는 과정을 몰랐다. 즉 <strong>단계마다 어떤 질문이 필요한 지 몰랐다</strong>. 그동안 DFS 문제를 꽤 많이 풀었는데 여전히 어렵고, 활용을 잘 못하는 이유도 아직 이 사고 방식에 낯설기 때문인 것 같다.</p><h3 class="mt-5 mb-3">두 번째 조건 - 탐색하기</h3><p>핵심 : 제한 안에서 적절히 탐색하기</p><p>O(n) 반복문 내부에서 숫자를 걸러내야 하기 때문에, filter와 같은 메서드는 사용이 안된다. 포인터 두개를 움직여 가며 log N으로 해결한다. 실제로 해시 값 배열을 조회하는 것이 아니기 때문에 가능하다.</p><p>앞선 DFS 사용과 마찬가지로, 이 경우에도 '적절한 도구 선택'이라는 질문까지 사고 과정이 놓여진다면, 문제가 꽤나 분명하게 풀렸을 것 같다.</p><hr class="my-5"><h2 class="mt-5 mb-3">결론</h2><p>알고리즘은 결국 문제 해결을 위한 <strong>도구</strong>이다. 단순히 해시를 공부하려 시작한 이 문제에서 dfs, 이진탐색까지 필요하게 될 지 몰랐다. 하지만 이 도구들은 <strong>해시를 구성하기 위해 사용할 수 있다</strong>. 해시는 키를 함수로 변환하여 값이 대응되게 저장하는 것이다. 문제가 요구하는 것은, <strong>해시를 어떻게 설계할 것</strong>인가-다. 이 과정에서 함수 인자의 형태, 조건, 타입을 단서로 적절한 알고리즘을 사용해 해시에 저장하는 것이 핵심이다.</p><p>'어차피 JS 적절한 객체를 잘 쓰는 거 아닌가? 굳이 해시 개념이라는 게 필요하나?'라는 의문이 해소됐다.</p>1e:[["$","$L1f",null,{"content":"$20"}],"$L21"]

</code></pre><p></p><p>이 과정에서 폐기된 접근법들은,</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p>info, query를 일반 객체로 만들고 비교<br>info[i], query[i]를 객체를 만들고 현재 조건 query[i]의 유효한 필드를 Object.entries, values로 탐색<br>query에서 O(n)으로 탐색이 끝나야 하기 때문에, 배열과 배열 요소 비교는 반복이 추가된다. 폐기</p></li><li><p>query와 info를 문자 부분만 따서 비교<br>배열과 배열 비교 O(n^2) 방법이므로 폐기.</p></li><li><p>Map에 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code> 문자열을 필드로, 점수를 숫자 배열로<br>이 경우 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 문자열이 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>의 부분이어도 찾아내야 하기 때문에 탐색이 안된다.</p></li></ol><h3 class="mt-5 mb-3" id="-4">한계와 오류들</h3><p>사고 과정 오류: Map(해시) 키-값의 설계 오류<br>집중해야 할 문제 : 2개의 조건과 배열, 2번의 탐색<br>핵심 문제: 탐색 최적화 - 어떻게 탐색을 두 번 최적화할 것인가?</p><p>사고 과정을 노트로 옮기고 질답을 작성하니 생각의 오류가 분명히 보인다. 특히 핵심 문제를 논리 전개 이전에 의식하지 못한 것이 연속된 실수로 이어졌다.</p><p>꼬이기 시작한 지점은 <strong>A9</strong>부터다. 키를 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 점수로 사용하는 것. 여기서 초기 추론과 완전히 반대의 선택을 내리기 시작했다.</p><ul class="tight" data-tight="true"><li><p>'점수'만 숫자 타입이므로, 어쩐지 중요한 역할을 할 거라 생각했다.</p></li><li><p>해시를 만들어야 하는데, 키로 <strong>필터링</strong>해서 여러 요소를 가져와야한다.</p></li><li><p>'점수 필터링'이 <strong>가장 중요</strong>하니 <strong>중간 연결고리로 쓰고</strong>, 값에는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>의 문자부분만 갖는 Set을 배열로 저장한다.</p></li><li><p>Set에는 부분집합을 검증하는 메서드 isSupersetOf, isSubsetOf가 있다.</p></li><li><p>쿼리 키(문자열 부분)가 부분집합임을 O(1)로 확인할 수 있을 것 같다.</p></li><li><p>이후 O(n) 순회에서 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 점수만으로 O(1) 접근하고, 문자열 부분을 "-", "and" 제거한 단어 Set으로 만들고 부분집합 여부로 갯수를 확인.</p></li><li><p>하지만 Set과 Set 배열의 비교이므로, <strong>여전히 최악의 경우 O(n^2)가 됨</strong></p></li></ul><p><strong>A9의 잘못된 가정</strong>에서 시작하니, 배열에 Set을 넣어 O(n) 탐색이 해결하려는 실수를 했다. "2. 첫번째 조건은 적절한 키로 해당하는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i~n]</code> 요소들을 얻음"이라고 초기 추론에서 분명 인지했는데, Set이면 해결될 거라 믿고, 잘못된 방법을 고치려는 데 시간을 너무 썼다.</p><p></p><p><strong>어떻게 키-값을 탐색에 유리하게 만드는가?</strong></p><p>처음엔 단순히 두 객체 배열에서 탐색 영역에 조건 해당하는 갯수 새기의 문제라고 여겼다. 하지만 이 문제에서 조건도 여러 개고, 탐색 대상에 해당하는 요소도 여러 개다. 즉 O(1)으로 배열을 먼저 얻고, 그 값들이 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>로 필터링하는 과정을 다시 최적화 해야한다. 두 단계의 최적화가 필요함을 제대로 인지하지 못했다.</p><p></p><p>추론 과정과 뒤집어진 결정을 내린 원인은 시간 압박이다. 어떤 구조의 해시 키-값이 맞지?라고 지나치게 고민하다가 시간 압박에 못이겨 내린 최선의 선택이 오류였던 것이다.</p><p><strong>여기서 배운 점은 막혔을 때 질문을 하는 것이다.</strong> 테이블을 돌려 지금 막힌 지점에서 유효한 질문을 던지면, 한번에 답에 이르지 않지만, 적어도 한 걸음은 가까워진다. 연쇄된 유효한 질문이 맞는 답으로 데려다 준다. 매번 반복되는 사고 막힘에 대처하는 구체적인 방법은 1.단서를 찾고, 2.거기서로부터 작은 질문과 작은 답을 연속하는 거다.</p><hr class="my-5"><h2 class="mt-5 mb-3" id="-5">정답의 추론 과정</h2><p>결국 30분 동안 생각만 하다가 제한 시간이 모두 지났다. 여기서 더 고민해도 변하지 않는다. 바로 GPT에게 물어보고, 코드를 <strong><mark class="px-0.5 py-0.5/2 rounded text-white font-medium hover:animate-pulse" data-color="#aa3131" style="background-color: #aa3131; color: inherit">손으로 작성하며 논리 구조를 파악</mark></strong>한다. 그러곤 다시 추론 과정을 복기하여 무엇이 잘못된 생각이었는 지 짚어본다.</p><h3 class="mt-5 mb-3" id="-6">문제에서 단서 찾기</h3><h4 class="mt-5 mb-3">1. 자료 구조가 주는 힌트</h4><ul class="tight" data-tight="true"><li><p>info: 탐색 대상<br>직군, 경력 ..., 점수(숫자) 모두 not null</p></li><li><p>query: 탐색 조건, query[i]가 개별 조건의 키<br>직군, 경력...나머지 모두 nullable | <strong>숫자 not null</strong></p></li></ul><p>숫자가 not null인 점을 캐치한 건 좋았다.  여기서 조건은 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i][점수]</code> 에 해당하는 <strong>숫자 이상</strong>을 걸러내야 한다. 나머지 모든 문자열은 (부분) 일치가 조건이고, <strong>숫자</strong>는 <strong>이상</strong>이 조건이다.  <strong>탐색이 두번 필요하단 것을 자료 구조와 문제 조건이 이미 명시적으로 알려주고 있다.</strong></p><h4 class="mt-5 mb-3">2. 집중 할 문제</h4><blockquote><p>어떻게 탐색을 최적화 하는가?  > 어떻게 <strong>두 번의 탐색</strong>을 최적화하는가?</p></blockquote><p>추론 과정을 돌이켜 보니 <mark class="px-0.5 py-0.5/2 rounded text-white font-medium hover:animate-pulse" data-color="#f8b200" style="background-color: #f8b200; color: inherit">탐색 최적화 = "O(n)을 O(1)로 만든다"</mark>만 집중했다. 1번의 탐색 O(1)으로 여러 대상 가져오기는 Map의 값이 배열이 되지 않는 이상 불가능하다. 여기까진 맞게 접근했다. 그 이후 키와 값에 각각 어떤 것을 넣을까 고민했다. 여기서 필터링을 키로 직접 구현할 수 있는 형태-에서 질문이 멈췄다. 이 고민의 '<strong>목적</strong>'을 떠올리면 더 쉽게 다음 질문으로 이어질 수 있다.</p><p></p><p>Before:<br>해시 사용해서 O(n) -> O(1) 어떻게 탐색하지?<br>(탐색이 2번 필요하다는 가정을 인지하지 않았다.)</p><p>After:<br>목적</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p>O(n)에서 O(1)으로 배열을 가져온다. - 첫번째 탐색 최적화</p></li><li><p>이 배열을 최적화된 방법으로 다시 탐색한다. - 두 번째 탐색 최적화</p></li></ol><p>조건이 2개이고, 찾아야할 대상 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i~n]</code>도 여러 개니까 탐색을 2번 한다. 2번째 탐색이 필요한 걸 인지하고, <strong>Map의 값에 어떤 타입의 배열이 되어야 탐색에 유리할 지, 의미있는 질문을 할 수 있다.</strong> Set을 만드는 부분집합이 O(n^2)이 될 걸 알면서도, 키 필터링에 숫자를 쓰는 것 말고 다른 방법이 안 떠올랐다. 그러는 대신에 '필터링 하는 키'보다, <strong>두 번째 배열을 효과적으로 탐색한다</strong>-에 집중하면, 더 쉽게 추론했을 것이다.</p><p></p><p>고민하다가 key에 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i].점수</code>인 숫자를 넣었다. '필터링 하는 키'를 <strong>일단은 찾았으니까</strong>. <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 문자열을 조각내어 Map에 추가하는 방법은 도무지 떠오르지 않았다. <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code> 문자의 부분집합인 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 문자를 어떻게 키로 구현할 방법을 고민하지 않았다. 어렵게 느껴져서 자동적으로 회피했던 것 같다. 문제 조건에 분명 어긋나지만 쉬워보이는 길이 있다면, 이젠 고민도 없이 내려놓겠다.</p><h4 class="mt-5 mb-3">3. 타입이 힌트</h4><p></p><p>O(1)으로 해시에서 배열을 가져온다. <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>는 숫자와 문자열로 이루어진다.<br><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>가 갖는 조건은</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>의 문자열 일부 or 전체가 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>와 일치</p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>의 숫자(점수)가 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>의 숫자 이상</p></li></ol><p>정답에 가까운 자료구조를 한번에 얻을 고민을 하기 전에 이렇게 생각해보자.</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p>해시에서 가져온 배열 안에 숫자와 문자 중 어떤 게 탐색에 쉬울까?</p></li><li><p>이 때 문자를 가져오는 조건은 부분 일치이고, 숫자는  x 이상이다.</p></li></ol><p>이렇게 고민하니 당연히 숫자란 답이 나온다. 문자 탐색 조건이 생각보다 간단하지 않기 때문이다. .includes는 기본적으로 O(n)이라 복잡도가 높다. 거기에 문제에서 문자열 조건은 더 복잡하다.  <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>가 갖는 4개의 필드 중 하나라도 일치하는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code>를 찾아야 한다.</p><p>키를 점수로 하여 X 이상을 필터링하고, 문자는 Set과 부분집합으로 필터링한다는 막연한 해결법은 잘못된 접근이었다. 방향이 중요하다. 타입과 2개의 조건이 꽤 분명하게 단서를 주고 있다.</p><h3 class="mt-5 mb-3" id="-7">원리와 추론 과정</h3><p>오류를 인지하고 필요한 단서들을 찾았으니, 새로운 방법으로 다시 추론 해본다.</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal tight" data-tight="true"><li><p>두 배열에서 한 요소로 여러 요소를 가져와야 한다.</p></li><li><p>가져올 때 조건은 2개다. 문자의 부분 일치와 숫자의 특정 값 이상.</p></li><li><p>탐색 조건은 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query</code>이다. <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query</code>에서 O(n) 순회를 한다.</p></li><li><p>조건이 2개이니 탐색 과정을 두 단계로 나눠서 진행한다.</p></li><li><p>1단계 : <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query</code>를 O(n)으로 순회하면서 여러 요소를 가져와야 하니 O(1), 키-값 구조가 적절하다. 이 때 가져올 값이 여러 개이니 값은 배열이 되어야한다.</p></li><li><p>2단계 : 가져온 여러 요소를 2번째 조건에 맞춰 탐색한다. O(n)이 아닌 최적화 탐색을 찾는다.</p></li><li><p>어떻게 1단계를 가져올까?</p></li><li><p>더 구체적으로, 어떻게 키가 info를 탐색할 수 있게 해시를 만들까?</p></li><li><p>답을 얻기 위해 초점을 바꿔 단서를 찾는다.</p></li><li><p>해시에서 키로 값을 가져올 때, 두가지 조건 중 어떤 것을 적용하는 게 나을까?</p></li><li><p>각 조건을 키로 사용할 때 케이스를 비교한다.</p></li><li><p>숫자(점수)를 키로 사용: 값이 문자 배열이 된다. 문자 비교는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>가 부분집합이 되어야 한다. 값 배열을 모두 순회하는 방법 밖에 없다.</p></li><li><p>문자를 키로 사용: 배열에서 쿼리 점수 이상만 탐색하면 되니 문자보다 쉽다.</p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">{ [key: string // 선택된 쿼리 문자] : number[] // 지원자들의 점수}</code> 키와 값의 타입을 결정했다. 2번째 탐색에서 숫자가 유리하기 때문이다.</p></li><li><p>문자열 키의 조건은 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">info[i]</code> 문자 일부가 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">query[i]</code>
