에뮬레이터</p><p></p></blockquote><p></p><h3>1.1 가상화의 역사</h3><h4>가상화 기술 시작</h4><ul><li><p><strong>하이퍼바이저</strong>와 같은 <strong>가상화 지원 기술</strong>은 일괄 처리를 수행하는 컴퓨터에 여러 명의 사용자가 동시에 액세스 가능하게 하는 대부분의 기업이 물리 서버와 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">단일 벤더 IT 스택</code>을 사용하고 있었기 때문에 다른 <strong>벤더의 하드웨어에서 레거시 애플리케이션을 실행할 수 없음</strong></p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">가상화 Virtualization</code>를 통해 기업은 <strong>서버를 파티셔닝</strong>하고 <strong>여러 유형 및 버전의 운영 체제에서 레거시 애플리케이션을 실행</strong></p></li><li><p>서버를 더 <strong>효율적</strong>으로 사용해 구매, 셋업, 냉각, 유지관리와 관련된 <strong>비용 감소</strong></p></li></ul><h3>1.2 하이퍼바이저의 정의와 가상화 원리</h3><h4>하이퍼바이저(<em>Hypervisor</em>)</h4><ul><li><p>소프트웨어가 물리 리소스를 필요로 하는 가상 환경으로부터 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">&lt;u&gt;</code><strong>리소스를 분리</strong><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">&lt;/u&gt;</code></p></li><li><p>호스트 컴퓨터 1대에서 <strong>운영체제 다수를 동시에 실행</strong>하는 <strong>논리적 플랫폼</strong></p><ul><li><p>Platform = 운영체제가 실행될 수 있는 <strong>기반 환경</strong></p></li></ul></li><li><p><strong>하이퍼바이저</strong>는 노트북 등의 <strong>운영 체제에 배포하거나 서버 등의 하드웨어에 직접 설치</strong></p></li><li><p><strong>하이퍼바이저</strong>가 물리 <strong>리소스를 분할하여 가상 환경에서 사용</strong></p></li></ul><h4>하이퍼바이저 유형</h4><h5>Native - Type 1</h5><blockquote><p>정의: 하이퍼바이저가 운영체제 없이 <strong>하드웨어 위에 직접 설치되는 형태</strong></p></blockquote><ul><li><p>해당 <strong>하드웨어/베어메탈에 직접 설치</strong></p></li><li><p><strong>게스트 운영체제는 두 번째 수준으로 실행</strong></p></li><li><p>제품: <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Xen</code>, <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded transition-colors duration-300">Xen</code> : 오픈소스 기반의 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors 조사했습니다.</p><ol class="my-3 px-1 py-3 text-sm leading-6 bg-muted/30 border border-secondary rounded list-decimal"><li><p>하드웨어 (CPU, 메모리, 디스크)</p></li><li><p>호스트 운영체제 (예: Windows, macOS, Linux)</p></li><li><p>하이퍼바이저 프로그램 (VirtualBox, VMware Workstation, Parallels 등)</p></li><li><p>게스트 운영체제(Guest OS) (가상 머신 안에 설치된 OS)</p></li></ol><p>여기서 게스트 OS는<br><code class="mx-1 px-1 bg-muted 표현한다고 합니다.</p><h3>1.3 *컨테이너</h3><p>개념의 중요성에 대해 강조해주셨습니다. 도커가 널리 사용되면서 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">컨테이너=도커</code>라고 합니다.<br>개인적으로는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Containerd</code>, 컨테이너 런타임이 궁금하여 조금 더 알아보고 싶었습니다.</p><h4>컨테이너의 개념</h4><ul><li><p>컨테이너는 가상환경을 사용해 각 <strong>마이크로 서비스</strong>를 <strong>격리(isolate)</strong>하는 기술</p></li><li><p>컨테이너는 <strong>가상머신처럼 하드웨어를 전부 구현하지 않기 때문에</strong> <strong>매우 빠른 실행 가능</strong></p><ul><li><p>최근에 M3 맥에서 UTM을 활용하여 Windows OS를 실행했는데 상당히 무거웠습니다.</p></li><li><p>컨테이너는 도커, K8S의 환경에서 더 작은 단위로 격리되어 실행되어 빠른 것이 아닐까 예상했습니다.</p></li></ul></li></ul><h3>1.4 VM과 컨테이너의 성능 비교</h3><p>이 부분은 사실 많이 낯선 내용이었습니다.<br>그래서 주로 벤치마크에서 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Hypervisor</code> / <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Docker</code> / <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">K8S</code>의 성능 비교 및 자원 효율성 위주로 이해했습니다.</p><h4>PageRank 워크로드 실행 결과</h4><ul><li><p>PageRank는 <strong>웹페이지들</strong>의 중요도를 수치로 평가하는 알고리즘</p></li><li><p>실제로 PageRank 점수를 계산하려면, 수백만~수십억 개의 노드와 링크로 구성된 대규모 그래프에서 위와 같은 연산을 반복 수행해야 함</p></li><li><p>이 과정은 행렬 곱셈, 벡터 연산 등 대량의 수치 계산과 반복적인 갱신이 필요하므로, <strong>컴퓨팅 자원을 많이 소모</strong></p></li><li><p>컨테이너 환경은 VM 환경에 비해 CPU 자원을 더 효율적으로 활용하며, 대기(Wait) 상태가 거의 없어 리소스 낭비가 적고, 작업 완료 시간도 짧음</p></li><li><p>VM 환경은 인스턴스가 많아질수록 CPU 대기(Wait) 시간이 급격히 늘어나고, 전체 작업 시간이 길어짐. 이는 가상화 오버헤드와 리소스 경합 때문임</p></li></ul><h4>컨테이너와 VM, 베어메탈 벤치마크 결과</h4><ul><li><p>베어메탈(Native), 가상머신(KVM), 그리고 다양한 컨테이너 환경(Docker, LXC, RKT)에서 CPU, 메모리, 디스크 I/O, 네트워크 성능을 비교한 벤치마크 결과</p></li><li><p>VM(KVM)은 모든 항목에서 <strong>오버헤드로 인해 성능이 떨어진다</strong></p></li></ul><h3>1.5 컨테이너 격리 기술</h3><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors 프로세스 격리를 통해 "뷰"를 제어한다는 의미로 이해했습니다.</p></li></ul><p>네임스페이스를 통해 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">프로세스마다 격리</code>한다는 개념이 중요한 것같습니다.</p><h4>B. cgroup(컨트롤 그룹)</h4><ul><li><p>프로세스 그룹이 사용할 수 있는 <strong>시스템 자원(메모리, CPU, I/O, 네트워크 등)의 "양"을 제한하거나 모니터링</strong>하는 역할</p></li><li><p>cgroup은 자원의 '할당/제한/계량'을 담당하고, 네임스페이스는 '격리'를 담당</p></li><li><p>/cpu_mem_cg는 systemd와 cgroupfs와 같은 "cgroup 계층(hierarchy)"의 최상위 관리 주체를 나타냄</p></li></ul><p>하지만 리눅스 네임스페이스, cgroup을 이용하여 컨테이너를 구성하는 것에 <strong>난이도가 있었기에 단순화한 툴(Docker)</strong>을 사용하게 됩니다.</p><hr class="my-5"><h1>2. 도커(Docker)</h1><h2>2.1 도커 소개 (2013년)</h2><ul><li><p>컨테이너 기술의 사실상 표준</p></li><li><p>다양한 운영체제에서 사용 가능(리눅스, 윈도우, 구성 요소로 개발</p></li><li><p>Docker는 2013년에 처음 출시되었으며 컨테이너화된 애플리케이션을 구축하고 배포하기 위한 도구로 빠르게 널리 채택됨</p></li><li><p><strong>레지스트리</strong>에서 이미지 가져오기, 컨테이너 시작 및 중지와 같은 컨테이너 관리와 관련된 하위 수준 작업을 처리하기 위해 Docker의 일부로 개발</p></li></ul><h3>독립형 프로젝트로의 전환</h3><ul><li><p>2017년 Docker에서 독립형 프로젝트 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Graduated Project</code>로 containerd가 추출되어 <strong>CNCF(Cloud Native Computing Foundation)</strong> 프로젝트가 됨</p></li><li><p>그 이후로 containerd는 성능, 안정성 및 보안에 중점을 두고 크게 발전하고 개선</p></li><li><p>Containerd에는 Docker, Google 및 Red Hat과 같은 회사의 개발자를 포함하여 대규모의 활동적인 기여자 커뮤니티가 있음</p></li></ul><h3>CNCF 프로젝트로서의 성장</h3><p>강의에서 <a target="_blank" rel="noopener noreferrer nofollow" href="https://www.cncf.io/">CNCF</a>를 알려주셔서 좋았습니다.</p><ul><li><p>Containerd는 CNCF(Cloud Native Computing Foundation) 프로젝트가 되어 가시성을 높이고 더 큰 기여자 커뮤니티를 유치하는 데 도움</p></li></ul><h3>컨테이너의 진화</h3><ul><li><p>Docker에서 추출된 이후 containerd는 <strong>성능, 안정성 및 보안</strong>에 중점을 두고 크게 발전</p></li><li><p>2018년에 출시된 containerd 1.0은 프로젝트의 안정성과 성숙도에 중요한 이정표를 세움</p></li><li><p><strong>2019년 Kubernetes의 기본 컨테이너 런타임으로 containerd 채택</strong></p></li><li><p>2020년에 "졸업" 프로젝트로 CNCF Landscape에 containerd 포함</p></li></ul><h2>2.4 컨테이너 레지스트리**</h2><p>"레지스트리" 개념이 크게 와닿지 않았으나, "저장소" 개념으로 보니 이해가 수월했습니다.<br>특히 duration-300">K8S</code>의 성능 비교 및 자원 효율성 위주로 이해했습니다.</p><h4>PageRank 워크로드 실행 결과</h4><ul><li><p>PageRank는 <strong>웹페이지들</strong>의 중요도를 수치로 평가하는 알고리즘</p></li><li><p>실제로 PageRank 점수를 계산하려면, 수백만~수십억 개의 노드와 링크로 구성된 대규모 그래프에서 위와 같은 연산을 반복 수행해야 함</p></li><li><p>이 과정은 행렬 곱셈, 벡터 연산 등 대량의 수치 계산과 반복적인 갱신이 필요하므로, <strong>컴퓨팅 자원을 많이 소모</strong></p></li><li><p>컨테이너 환경은 VM 환경에 비해 CPU 자원을 더 효율적으로 활용하며, 대기(Wait) 상태가 거의 없어 리소스 낭비가 적고, 작업 완료 시간도 짧음</p></li><li><p>VM 환경은 인스턴스가 많아질수록 CPU 대기(Wait) 시간이 급격히 늘어나고, 전체 작업 시간이 길어짐. 이는 가상화 오버헤드와 리소스 경합 때문임</p></li></ul><h4>컨테이너와 VM, 베어메탈 벤치마크 결과</h4><ul><li><p>베어메탈(Native), 가상머신(KVM), 그리고 다양한 컨테이너 환경(Docker, LXC, RKT)에서 CPU, 메모리, 디스크 I/O, 네트워크 성능을 비교한 벤치마크 결과</p></li><li><p>VM(KVM)은 모든 항목에서 <strong>오버헤드로 인해 성능이 떨어진다</strong></p></li></ul><h3 id="1-5">1.5 컨테이너 격리 기술</h3><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Container</code>는 기본적으로 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Linux kernel</code> 기능을 통해 구현된다고 합니다.</p><h4>A. 리눅스 네임스페이스(Namespace)</h4><ul><li><p>각 프로세스가 파일 시스템 마운트, 네트워크, 유저(uid), 호스트 네임(uts) 등에 대해 <strong>시스템에 독립 뷰를 제공</strong></p></li><li><p>What you can see - 프로세스 격리를 통해 "뷰"를 제어한다는 의미로 이해했습니다.</p></li></ul><p>네임스페이스를 통해 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">프로세스마다 격리</code>한다는 개념이 중요한 것같습니다.</p><h4>B. cgroup(컨트롤 그룹)</h4><ul><li><p>프로세스 그룹이 사용할 수 있는 <strong>시스템 자원(메모리, CPU, I/O, 네트워크 등)의 "양"을 제한하거나 모니터링</strong>하는 역할</p></li><li><p>cgroup은 자원의 '할당/제한/계량'을 담당하고, 네임스페이스는 '격리'를 담당</p></li><li><p>/cpu_mem_cg는 systemd와 cgroupfs와 같은 "cgroup 계층(hierarchy)"의 최상위 관리 주체를 나타냄</p></li></ul><p>하지만 리눅스 네임스페이스, cgroup을 이용하여 컨테이너를 구성하는 것에 <strong>난이도가 있었기에 단순화한 툴(Docker)</strong>을 사용하게 됩니다.</p><hr class="my-5"><h1 id="2-docker">2. 도커(Docker)</h1><h2 id="2-1-2013">2.1 도커 소개 (2013년)</h2><ul><li><p>컨테이너 기술의 사실상 표준</p></li><li><p>다양한 운영체제에서 사용 가능(리눅스, 윈도우, MacOS)</p></li><li><p>앱App에 국한 되지 않고 <strong>의존성 및 파일 시스템까지 패키징하여 빌드, 배포, 실행을 단순화</strong></p></li><li><p>리눅스의 <strong>네임 스페이스와 cgroups와 같은 커널 기능</strong>을 사용하여 가상화</p></li></ul><p></p><h2 id="docker-daemon">도커 데몬 (Docker Daemon)</h3><ul><li><p>도커의 핵심 서비스이며, <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">컨테이너의 생성, 실행, 관리</code> 등의 작업을 수행</p></li><li><p>데몬은 컨테이너 실행에 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">필요한 리소스를 관리</code>하고 <strong>도커 API를 통해 클라이언트 요청에 응답</strong></p></li></ul><h3 id="containerd">컨테이너드* (Containerd)</h3><ul><li><p>도커 데몬에서 사용하는 <strong>핵심 컨테이너 런타임</strong></p></li><li><p>도커는 버전 1.11 이후로 <strong>컨테이너 런타임을 분리하여 컨테이너드</strong>를 사용</p></li><li><p>컨테이너드는 <strong>컨테이너의 생명주기 관리, 이미지 관리, 네트워킹, 스토리지</strong> 등과 관련된 기능을 담당</p></li><li><p>컨테이너를 관리하는 <strong>중간관리자의 역할</strong></p></li></ul><h3 id="runc">런타임 (Runc)</h3><ul><li><p><strong>컨테이너를 실행하는 데 사용되는 도구</strong></p></li><li><p>도커는 기본적으로 런타임으로 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Runc</code>는 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded 컨테이너의 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">생명주기 관리</code>와 관련된 핵심적인 작업을 담당</p></li></ul><h2 id="2-3-containerd">2.3 containerd의 역사</h2><h3 id="-1">개발 배경</h3><ul><li><p>Containerd는 원래 유명한 컨테이너 플랫폼인 Docker의 구성 요소로 개발</p></li><li><p>Docker는 2013년에 처음 출시되었으며 컨테이너화된 애플리케이션을 구축하고 배포하기 위한 도구로 빠르게 널리 채택됨</p></li><li><p><strong>레지스트리</strong>에서 이미지 가져오기, 컨테이너 시작 및 중지와 같은 컨테이너 관리와 관련된 하위 수준 작업을 처리하기 위해 Docker의 일부로 개발</p></li></ul><h3 id="-2">독립형 프로젝트로의 전환</h3><ul><li><p>2017년 Docker에서 독립형 프로젝트 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Graduated Project</code>로 containerd가 추출되어 <strong>CNCF(Cloud Native Computing Foundation)</strong> 프로젝트가 됨</p></li><li><p>그 이후로 containerd는 성능, 안정성 및 보안에 중점을 두고 크게 발전하고 개선</p></li><li><p>Containerd에는 Docker, Google 및 Red Hat과 같은 회사의 개발자를 포함하여 대규모의 활동적인 기여자 커뮤니티가 있음</p></li></ul><h3 id="cncf">CNCF 프로젝트로서의 성장</h3><p>강의에서 <a rel="noopener noreferrer nofollow" href="https://www.cncf.io/">CNCF</a>를 알려주셔서 좋았습니다.</p><ul><li><p>Containerd는 CNCF(Cloud Native Computing Foundation) 프로젝트가 되어 가시성을 높이고 더 큰 기여자 커뮤니티를 유치하는 데 도움</p></li></ul><h3 id="-3">컨테이너의 진화</h3><ul><li><p>Docker에서 추출된 이후 containerd는 <strong>성능, 안정성 및 보안</strong>에 중점을 두고 크게 발전</p></li><li><p>2018년에 출시된 containerd 1.0은 프로젝트의 안정성과 성숙도에 중요한 이정표를 세움</p></li><li><p><strong>2019년 Kubernetes의 기본 컨테이너 런타임으로 containerd 채택</strong></p></li><li><p>2020년에 "졸업" 프로젝트로 CNCF Landscape에 containerd 포함</p></li></ul><h2 id="2-4">2.4 컨테이너 레지스트리**</h2><p>"레지스트리" 개념이 크게 와닿지 않았으나, "저장소" 개념으로 보니 이해가 수월했습니다.<br>특히 강의에서 이미지, 컨테이너, 레지스트리 각각의 역할을 구분하여 숙지하면 좋습니다.</p><h3 id="-4">기본 개념</h3><ul><li><p><strong>컨테이너</strong>: 이미지를 격리하여 독립된 공간에서 실행한 가상 환경(ENV)</p></li><li><p><strong>이미지</strong>: 필요한 프로그램과 라이브러리, 소스를 설치한 뒤 만든 하나의 파일</p><ul><li><p><em>이미지로 말아 올리다</em>라는 표현을 자주 쓰며, 의존성, 환경 파일 등을 하나의 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">.tar</code> 파일로 압축한 것을 도커 <code On-Premises : Hosted Docker Registry</p><ul><li><p>폐쇄망/내부 네트워크를 구성하여 올린 레지스트리입니다.</p></li><li><p>반드시 적절한 네트워크와 함께 레지스트리가 갖추어져야 도커를 사용할 수 있습니다.</p></li><li><p><code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">Harbor</code> CI/CD 관련 내용에서 더 살펴볼 예정입니다.<br>b. Public Cloud:</p></li><li><p>Docker Trusted Registry, Azure Container, AWS, Google 등 공공 클라우드에 해당합니다.</p></li></ul></li><li><p>누구나 이미지를 만들어 푸시할 수 있으며 푸시된 이미지는 다른 사람들에게 공유 가능</p></li></ul><h2 id="2-5">2.5 도커의 한계</h2><ul><li><p>서비스가 커지면 커질 수록 관리해야 하는 컨테이너의 양이 급격히 증가</p></li><li><p>도커를 사용하여 관리를 한다 하더라도 쉽지 않은 형태</p></li></ul><p>예시:</p><ul><li><p>32대의 서버의 호스트에 도커가 설치되었다고 관리의 문제를 해결하기 위해 도입되었습니다.도커 레이어를 건너 뛰고, 모든 컨테이너를 <code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">K8S</code> 플러그인을 통해 관리하게 됩니다.</p><ul><li><p>2014년 구글이 오픈 소스 공개</p></li><li><p>구글이 컨테이너 운영 노하우가 담긴 오픈소스</p></li><li><p>고대 그리스어로 항해사라는 의미를 가짐</p></li><li><p>다수의 컨테이너를 자동으로 운영하기 위한 <strong>오케스트레이션 도구</strong></p></li><li><p>많은 시스템을 통합, 컨테이너를 다루기 위한 API 제공</p></li></ul><h2 id="3-2">3.2 이미지, 컨테이너, 도커와 쿠버네티스의 관계</h2><p>마지막으로 정리하면 다음과 같습니다.</p><ul><li><p><strong>컨테이너</strong>: 용기 = 리눅스 커널(<code class="mx-1 px-1 bg-muted dark:bg-muted-foreground rounded text-orange-700 dark:text-theme italic font-mono transition-colors duration-300">name_space+cgroup</code>)로 구성. <code Container(용기)로 이해.</p></li><li><p><strong>도커</strong>: 항만 노동자</p></li><li><p>개별 컨테이너를 관리하는 관리자</p></li><li><p><strong>쿠버네티스</strong>: 항해사</p></li><li><p>모든 컨테이너를 관장하는 관리자</p></li></ul><hr
