REST에 너무 익숙해져 비동기 함수는 그 방식 밖에 떠올리지 못하는 게 오히려 gql에 새로 적응하는데 방해가 되었기 때문이다. GQL은 REST와 확실히 다르다.</p><p><br></p><p><img src="http://res.cloudinary.com/dolziw8fv/image/upload/v1666163858/qkdobnmzlayxpswrgnom.jpg"></p><p><em>gql로 구현한 페이지네이션에서 나타나는 동일한 endpoint의 post 요청들.</em></p><p><br></p><ul><li><strong>endpoint 최소화의 장점</strong></li></ul><p><br></p><p>직접 사용해보니 <strong>endpoint의 최소화가 가져오는 이점이 상당하다.</strong> 먼저 클라이언트에서 <u>일일히 api에 대응하는 비동기 함수를 작성할 일이 확실히 적어진다.</u> 혹은 여러 응답 <u>데이터를 합성하거나 가공할 일이 거의 없어진다</u>. 이와 관련해서 overfetching/underfetching 이슈들이 자주 언급된다.</p><p><br></p><ul><li>overfetching : 클라이언트에서 필요 이상의 응답 데이터를 받게 된다. id, name 정도만 필요한 요청에도 수많은 필드가 함께 오는 경우가 잦다.</li><li>underfetching : api가 필요한 값을 제공하지 않아 여러 api를 호출하고 클라이언트에서 데이터를 가공해야한다.</li></ul><p><br></p><p>두 이슈는 프론트 개발에서 지루하고 반복적이지만 피할 수 없는 코드를 발생시킬 때가 많다. rest api를 사용하는 전직장에선 오버/언더 페칭은 피할 수 없는 숙명처럼 자주 접했다. 모든 api가 그렇진 않지만, overfetch된 한 api에서 필요한 작은 부분들을 따서 개발을 하거나, overfetch된 여러 api에서 다시 작은 부분을 따와서 하나로 묶어 사용할 때가 잦았다(underfetch). 즉 응답 이후에도 데이터 포맷팅에 많은 시간을 들이게 된다.</p><p><br></p><p>이 문제는 결과적으로<strong> 너무 많은 시간 소요를 가져온다</strong>. 응답 데이터를 가공하면서, api 수정을 요청하면서, api 개발을 기다리면서, 기다리는 동안 작성한 코드를 수정하면서. <u>모든 단계에서 소요가 발생한다.</u> 이는 프론트에서 편리한 데이터 상태관리 툴을 사용하는 것과도 무관했다. 또한 api docs 없이 응답 데이터를 예측하기가 어렵다. 데이터 커플링 때문에 프론트의 코드 수정이 잦아지고, api 수정에 매우 민감하기 때문에 코드 수명이 짧아지는 원인이 되지 않았을까도 싶다. 결과적으로 너무 효율적이지 못하다. graphql은 이러한 문제를 상당 부분 개선할 수 있다.</p><p><br></p><ul><li><strong>타입과 필드의 장점</strong></li></ul><p><br></p><p><u>GQL은 요청/응답의 데이터의 타입을 </u><strong><u>Schema</u></strong><u>를 통해 지정한다</u>. 프레임워크마다 차이가 있겠지만, 내가 사용한 <strong><u>Relay</u></strong>는 schema.graphql에 작성된 쿼리의 타입을 (artifact를 통해)그대로 가져올 수 있었다. 즉 프론트에서 <strong>응답 데이터의 타입을 자동으로 지정할 수 있다</strong>. endpoint 최소화에 이어 가히 혁신적이라 할 수 있다. 특히 타입스크립트를 사용한다면, interface와 type을 줄이는 것만으로도 상당량의 작업 시간과 코드를 줄일 수 있다.</p><p><br></p><p><br></p><ul><li><strong>쿼리의 재사용, 유지보수의 간편함</strong></li></ul><p><br></p><p>"<em style="color: rgb(32, 32, 32);">GraphQL은 특정 데이터베이스에 제한받지 않고</em><span style="color: rgb(32, 32, 32);">"</span> 동일한 api를 제공한다. 만일 클라이언트에서 응답에 필드를 하나 추가하여 받고자 한다면, 쿼리에 필드 한 줄만 추가해주면 된다.</p><pre class="ql-syntax" spellcheck="false">query {&nbsp;

&nbsp; &nbsp; login
    + url  // url을 추가로 요청한다

}

</pre><p>프레임워크마다 차이가 있겠지만, 굳이 fragment가 아니더라도 query를 import하여 재사용할 수도 있다. api마다 다른 비동기 함수와 header, body를 작성해야 하는 지긋지긋한 반복 작업에서 벗어날 수 있다. 또한 gql의 query는 상당히 직관적이기 때문에 어떤 데이터를 요청하고 반환 받을지 예측하기도 쉽다.</p><p><br></p><p>이외에도 gql의 이점이 상당히 많다. 그 중 특히 캐싱이 상당히 인상깊다. 관련해서는 <a href="https://blog.cometkim.kr/posts/thinking-in-graphql-ko/#%EA%B7%B8%EB%9E%98%ED%94%84-%EC%BA%90%EC%8B%B1%ED%95%98%EA%B8%B0" rel="noopener noreferrer" target="_blank">Thinking in GraphQL의 캐싱 내용을 읽어보시길 추천드린다.</a></p><p><br></p><h2>GraphQL의 단점</h2><p><br></p><p>아직 gql을 깊게 사용해본 것은 아니기에 정확히 구체적으로 파악하고 있지는 못하다. 그리고 클라이언트에서만 사용해 본 한계가 있다. 그럼에도 불구하고 몇 가지를 생각해보자면..</p><p><br></p><ol><li>진입 장벽이 있다. </li><li>서버에서 작성된 스키마를 파악하는데 시간이 걸린다.</li></ol><p><br></p><p>1)은 사실 gql 보다는 gql 프레임워크와 연관이 깊다. 관련해서 릴레이 문서에서 더 자세히 다뤄보기로 하겠다. 2)은 사실 github graphql api를 사용하면서 느낀 점이다. 1:n, n:n 테이블에 있는 데이터를 가져오기 위해 connection을 파악하고, inline fragment를 사용해야 한다. 문법의 난이도보다도, 그 과정에서 github api에서 스키마를 파악하는 것이 까다롭게 느껴졌다. api가 하나이다 보니, 스키마가 방대해지면 쿼리를 작성하기 까다로울 수 있을 것 같다.</p><p><br></p><p><br></p><h2>GraphQL의 주요 요소들</h2><p><br></p><p><strong>Schema &amp; Resolver</strong></p><pre class="ql-syntax" spellcheck="false">// schema.graphql

type Character {
  name: String!
  appearsIn: [Episode!]!
}

  id: ID!
  name: String!
  length(unit: LengthUnit = METER): Float
}

  hero(episode: Episode): Character
  starship(id: ID!): Starship 
}
</pre><p>Schema : query, mutation 등 우리가 보낼 요청과 응답의 타입으로 구성된다. 그렇기에 우리는 임의의 argument를 아무 필드에 적용시킬 수 없고, 타입에 일치하지 않는 값을 argument로 넘겨줄 수도 없다. 스키마는 일종의 설계도로 기능하고, gql 프레임워크 컴파일러가 필드를 인식할 수 있게 도와준다.</p><p><br></p><p><br></p><pre class="ql-syntax" spellcheck="false">const resolver = {
Query {
 hero: ( __, episode) =&gt; getCharacter(episode),
 starship: (__, id) =&gt; getStarship(id)
}

}
</pre><p>Resolver : schema에서 지정된 타입을 바탕으로 query, mutation, fragment 등을 실제로 구현해주는 함수로 구성되어있다. 스키마에서 정의된 모든 필드를 리졸버에서 실제로 구현하는 것으로 보인다. resolver는 db로 요청을 보내는 실질적인 api 부분으로 이해했다. (MVC 패턴의 서비스-모델 부분에 해당하겠다.) 클라이언트에서는 스키마와 리졸버로 구현된 gql api의 타입에 맞춰 gql query를 작성하게 된다.</p><p><br></p><p><strong>Query &amp; Argument</strong></p><pre class="ql-syntax" spellcheck="false">query ExampleQuery {

REST에 너무 익숙해져 비동기 함수는 그 방식 밖에 떠올리지 못하는 게 오히려 gql에 새로 적응하는데 방해가 되었기 때문이다. GQL은 REST와 확실히 다르다.</p><p><br></p><p><img alt="img" width="100%" height="100%" src="https://res.cloudinary.com/dolziw8fv/image/upload/v1666163858/qkdobnmzlayxpswrgnom.jpg"></p><p><em>gql로 구현한 페이지네이션에서 나타나는 동일한 endpoint의 post 요청들.</em></p><p><br></p><ul><li><strong>endpoint 최소화의 장점</strong></li></ul><p><br></p><p>직접 사용해보니 <strong>endpoint의 최소화가 가져오는 이점이 상당하다.</strong> 먼저 클라이언트에서 <u>일일히 api에 대응하는 비동기 함수를 작성할 일이 확실히 적어진다.</u> 혹은 여러 응답 <u>데이터를 합성하거나 가공할 일이 거의 없어진다</u>. 이와 관련해서 overfetching/underfetching 이슈들이 자주 언급된다.</p><p><br></p><ul><li>overfetching : 클라이언트에서 필요 이상의 응답 데이터를 받게 된다. id, name 정도만 필요한 요청에도 수많은 필드가 함께 오는 경우가 잦다.</li><li>underfetching : api가 필요한 값을 제공하지 않아 여러 api를 호출하고 클라이언트에서 데이터를 가공해야한다.</li></ul><p><br></p><p>두 이슈는 프론트 개발에서 지루하고 반복적이지만 피할 수 없는 코드를 발생시킬 때가 많다. rest api를 사용하는 전직장에선 오버/언더 페칭은 피할 수 없는 숙명처럼 자주 접했다. 모든 api가 그렇진 않지만, overfetch된 한 api에서 필요한 작은 부분들을 따서 개발을 하거나, overfetch된 여러 api에서 다시 작은 부분을 따와서 하나로 묶어 사용할 때가 잦았다(underfetch). 즉 응답 이후에도 데이터 포맷팅에 많은 시간을 들이게 된다.</p><p><br></p><p>이 문제는 결과적으로<strong> 너무 많은 시간 소요를 가져온다</strong>. 응답 데이터를 가공하면서, api 수정을 요청하면서, api 개발을 기다리면서, 기다리는 동안 작성한 코드를 수정하면서. <u>모든 단계에서 소요가 발생한다.</u> 이는 프론트에서 편리한 데이터 상태관리 툴을 사용하는 것과도 무관했다. 또한 api docs 없이 응답 데이터를 예측하기가 어렵다. 데이터 커플링 때문에 프론트의 코드 수정이 잦아지고, api 수정에 매우 민감하기 때문에 코드 수명이 짧아지는 원인이 되지 않았을까도 싶다. 결과적으로 너무 효율적이지 못하다. graphql은 이러한 문제를 상당 부분 개선할 수 있다.</p><p><br></p><ul><li><strong>타입과 필드의 장점</strong></li></ul><p><br></p><p><u>GQL은 요청/응답의 데이터의 타입을 </u><strong><u>Schema</u></strong><u>를 통해 지정한다</u>. 프레임워크마다 차이가 있겠지만, 내가 사용한 <strong><u>Relay</u></strong>는 schema.graphql에 작성된 쿼리의 타입을 (artifact를 통해)그대로 가져올 수 있었다. 즉 프론트에서 <strong>응답 데이터의 타입을 자동으로 지정할 수 있다</strong>. endpoint 최소화에 이어 가히 혁신적이라 할 수 있다. 특히 타입스크립트를 사용한다면,
