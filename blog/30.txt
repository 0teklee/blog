(프로그래밍에서 가장 어려운 것 중 하나)</li><li>같은 데이터를 요청하는 다중 요청을 하나의 요청으로 병합하기</li><li>데이터가 "오래된" 것인지 알 수 있는 방법</li><li>데이터 업데이트를 가능한 한 빠르게 반영하기</li><li>페이지네이션, 데이터 늦은 로딩 등의 성능 최적화</li><li>서버 상태의 메모리 및 가비지 컬렉션 관리</li><li>구조적 공유를 이용한 쿼리 결과 memoization</li></ul><p><br></p><p>react-query는 이러한 서버 상태를 처리하는데 특화되어 있고, 이러한 문제들을 해결하는데 도움을 줄 수 있습니다. 이를 비교할 경우,<strong><u>&nbsp;SWR은 주로 브라우저 캐싱을 통해 클라이언트 상태를 관리하는데 초점을 맞추고 있지만, react-query는 서버 상태를 관리하는데 초점을 맞추고 있습니다.&nbsp;</u>따라서, 서버 상태를 처리하는데 초점을 맞추어야 하는 경우 react-query가 더 적합할 수 있습니다.</strong></p><p><br></p><p><br></p><p>비동기 처리에 대한 문제 정의와 라이브러리가 필요한 이유를 깊이 있고 명쾌하게 설명해주고 있다. 서버 상태에 초점을 맞춘다는 점이 인상깊다.</p><p><br></p><p><br></p><h3>SWR vs React query 사용 후기</h3><p><br></p><p>두 라이브러리 모두 사용하고 느낀점. swr은 자동으로 해주는 부분이 많아서 편하다. 편하기 때문에 깊게 알 필요가 없을 때가 많다. 즉 손가는 대로 사용하기 쉽다. 반면 리액트 쿼리는 직접 손 댈 여지가 많다. 물론 마찬가지로 생각없이 사용하기 편하나, 더 나은 방식이 아닐 가능성이 크다. 다만 제대로 알고 사용하기까지 swr보다 난이도가 좀 있는 편인 것 같다.</p><p><br></p><p>그리고 캐싱을 제어하는 옵션, 메소드들이 상당히 많다. 성능 최적화에 핵심이 될 수 있겠다고 얘기한 이유는 그 때문이다. 리액트 쿼리를 사용하다 보면 캐싱의 중요성을 느낄 수 밖에 없게 된다.</p><p><br></p><p><br></p><h3>QueryClient, useQuery</h3><p><br></p><p>쿼리 클라이언트는 "캐시와 상호작용하기 위해 사용한다". 개인적으로 queryClient의 캐싱이 리액트 쿼리의 근간처럼 느껴졌다. 쿼리 클라이언트를 사용하기 위해선, QueryClientProvider로 쿼리가 사용될 컴포넌트를 감싸주어야한다. 아래 예시는 블로그 방명록에 사용한 코드.</p><p><br></p><pre class="ql-syntax" spellcheck="false">&lt;QueryClientProvider client={queryClient}&gt;

&lt;/QueryClientProvider&gt;

</pre><p>클라이언트에 연결된 mutation 캐시를 반환하는 메소드이다.</p><p><br></p><p><br></p><p><br></p><ul><li><a href="https://tanstack.com/query/v4/docs/react/reference/QueryClient#queryclientinvalidatequeries" rel="noopener noreferrer" target="_blank" style="color: inherit;">queryClient.invalidateQueries</a></li></ul><pre class="ql-syntax" spellcheck="false">await queryClient.invalidateQueries({

}, { throwOnError, cancelRefetch })
