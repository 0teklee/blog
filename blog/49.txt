HTTP 통신이 전송 중 파괴, 증복, 왜곡되는 걱정을 하지 않아도 된다. 개발자는 인터넷의 결함이나 약점에 대한 걱정 없이 애플리케이션 고유의 기능 구현에 집중할 수있다.</p><h3><br></h3><p><br></p><h3>2. 웹 서버와 웹 클라이언트</h3><blockquote>HTTP 클라이언트와 HTTP 서버는 월드 와이드 웹의 기본요소이다.</blockquote><p><br></p><p><strong>웹 서버</strong></p><ul><li>데이터를 저장하고 HTTP 클라이언트가 요청한 데이터를 응답으로 돌려준다</li><li>웹 컨텐츠(리소스)는 웹 서버에 존재</li><li>http 프로토콜로 의사소통하기에 보통 HTTP 서버로 불림</li></ul><p><br></p><p><strong>웹 클라이언트</strong></p><ul><li>대표적으로 웹 브라우저가 있다.</li><li>ex) 브라우저로&nbsp;<a href="http://www.oreilly.com/index.html" rel="noopener noreferrer" target="_blank" style="color: var(--link-external-color);">http://www.oreilly.com/index.html</a>&nbsp;페이지를 열면, 브라우저는 HTTP 요청을&nbsp;<a href="http://www.oreilly.com/" rel="noopener noreferrer" target="_blank" style="color: var(--link-external-color);">www.oreilly.com</a>&nbsp;서버로 보낸다.</li></ul><h3><br></h3><p><br></p><h3>3. 리소스</h3><blockquote>1. 미디어 타입 MIME 2. URI</blockquote><p><br></p><p>웹 서버는 리소스를 관리/제공한다. 웹 리소스가 웹 컨텐츠의 원천이다.</p><p><br></p><ul><li>정적 리소스 : 텍스트, html, 워드, pdf, jpeg, avi 그 외 모든 종류의 파일을 포함</li><li>동적 리소스: 요청에 따라 콘텐츠를 생산하는 프로그램이 될 수 있음. ex) 사용자, 시간, 어떤 정보를 요청했는 지에 따라 변할 수 있는 리소스. 주식 거래, 라이브 영상, 부동산 데이터 베이스 검색, 쇼핑몰에서 선물 구입 등..</li></ul><p><br></p><p>결과적으로 어떠한 컨텐츠 소스도 리소스가 될 수 있다. (그렇기에 미디어 타입을 나눈다)</p><h4><br></h4><h4>3-1 미디어 타입</h4><p><br></p><p><strong style="color: var(--bold-color);">MIME(Multipurpose Internet Mail Extensions) 타입:</strong></p><p>HTTP는 웹에서 전송되는 객체 각각에 MIME 타입이라는&nbsp;<span style="color: var(--bold-color);">데이터 포맷 라벨</span>을 붙인다. 앞에서 살펴보듯이, 인터넷은 수천 가지 데이터 타입을 다루기 때문이다.</p><p><br></p><ul><li><span style="color: var(--bold-color);">웹 브라우저</span>는 서버로부터 받는 객체를 MIME 타입을 통해 확인</li><li>이를 통해 이미지 파일을 보여주거나, html 파일을 분석/포맷팅, 오디오 파일 재생, 특별한 포맷을 다루기 위해 외부 플러그인을 실행함</li><li>HTTP 헤더의&nbsp;<span style="color: var(--code-normal); background-color: var(--code-background);">Content-Type</span>에 명시한다.</li><li>타입은 여러가지가 있으나 "primary object type"/"specific subtype" 형식으로 표기한다.</li><li>HTML: text/html</li><li>plain ASCII 텍스트: text/plain</li><li>JPEG 이미지: image/jpeg</li><li>GIF 이미지: image/gif</li><li>PDF 문서 : application/pdf</li><li>etc..</li></ul><h4><br></h4><h4>3-2 URI</h4><p>웹 서버에 있는 리소스는 각자 이름(주소)를 갖고 있고, 클라이언트는 이를 지목하여 리소스를 요청할 수 있다. 개별 서버 리소스를 URI(uniform resource identifier)라고 부른다. URI는 인터넷 상의 자원(리소스)을 고유하게 식별하고 위치를 지정하는 표준화된 방법을 제공한다. 서버 상의 각 파일, 문서, 이미지 등의 리소스는 고유한 URI를 가지며, 이를 통해 해당 리소스에 접근할 수 있다.</p><p><br></p><p><span style="color: var(--bold-color);">URL</span></p><p>URL은 리소스가 위치하는 리소스의 주소를 구체적으로 지정하는 형태다. 주로 3가지로 구성된다.</p><p><br></p><ul><li>스킴 : 프로토콜</li><li>주소 : 서버의 인터넷 주소</li><li>리소스 : 리소스의 위치</li></ul><p><br></p><pre class="ql-syntax" spellcheck="false">http://www.example.com/sample/sample_image.jpeg

background-color: var(--code-background);">ftp</span>&nbsp;등 여러 프로토콜이 있음.</li><li><span style="color: var(--bold-color);">호스트(host)</span>: 리소스가 호스팅되는 서버의 도메인 이름, IP 주소</li><li><span style="color: var(--bold-color);">포트(port)</span>: 서버에서 리소스에 접근하기 위해 사용되는 포트 번호. 일반적으로는 생략되며, 생략된 경우 기본 포트가 사용됨 (예: HTTP의 경우 80, HTTPS의 경우 443).</li><li><span style="color: var(--bold-color);">경로(path)</span>: 서버 내에서 리소스의 위치</li><li><span style="color: var(--bold-color);">쿼리(query)</span>: 선택적으로 리소스에 대한 추가적인 질의를 제공합니다.&nbsp;<span style="color: var(--code-normal); background-color: var(--code-background);">?</span>로 시작하며,&nbsp;<span style="color: var(--code-normal); background-color: var(--code-background);">키=값</span>&nbsp;쌍의 형태로 정보를 전달</li><li><span style="color: var(--bold-color);">프래그먼트(fragment)</span>: 리소스 내의 특정 부분을 가리키는 선택적인 구성 요소.&nbsp;<span style="color: var(--code-normal); background-color: var(--code-background);">#</span>로 시작.</li></ul><p><br></p><p>URN</p><p>리소스의 위치와 무관하게 리소스를 고유하게 식별하는 URI(Uniform Resource Identifier)의 한 형태. 리소스 위치 대신, 이름을 기반으로 그 자체의 고유성 제공. 예를 들어 RFC(Request for Comments), ISBN(International Standard Book Number) 등이 있다. ex)<em>&nbsp;urn:ietf:rfc:2141</em></p><h3><br></h3><p><br></p><h3>4. 트랜잭션</h3><blockquote>클라이언트가 웹 서버와 리소스를 주고 받기 위해 HTTP를 사용하는 법</blockquote><p><br></p><p><strong style="color: var(--bold-color);">HTTP 트랜잭션</strong></p><p><br></p><ul><li>요청 명령과 응답 결과로 구성</li><li>요청/응답의 상호작용은 HTTP 메시지라는 데이터 덩어리를 통해 이루어짐</li><li>요청 메시지, 응답 메시지를 통해 요청/응답의 트랜젝션</li><li>ex) 한 개의 웹페이지는 여러 HTTP 트랜젝션을 수행한다</li><li class="ql-indent-1">html을 요청하고 응답을 받는다</li><li class="ql-indent-1">html에서 link, script 등의 태그를 통해 이미지, 스타일, js 스크립트 등 여러 리소스를 요청한다.</li><li class="ql-indent-1">대부분의 경우 하나의 웹페이지는 하나의 리소스가 아닌 리소스의 모음이다.</li></ul><h3><br></h3><p><br></p><h3>5. HTTP 메시지</h3><p><br></p><p>각 소항목의 내용들은 Chapter 3에서 더 자세히 다룰 예정이다. 개괄적으로 보면 다음과 같다.</p><p><br></p><ul><li>HTTP 메시지는 단순한 줄 단위의 문자열이다.</li><li>요청, 응답 메시지 이외에 HTTP 메시지는 없다.</li><li>요청 메시지: 클라이언트에서 서버로 보내는 메시지</li><li>응답 메시지: 서버에서 클라이언트로 보내는 메시지</li></ul><p><br></p><p>HTTP 메시지는 3가지로 구성된다.</p><h4><br></h4><h4>1. 시작줄</h4><p>메시지의 첫 줄에 해당하며, 요청/응답은 다음과 같이 구성된다.</p><p><br></p><pre class="ql-syntax" spellcheck="false">`GET /index.html HTTP/1.1` 
</pre><p>요청 메시지 : HTTP 메서드 / 요청 대상(URI) / HTTP 버전</p><pre class="ql-syntax" spellcheck="false">- `HTTP/1.1 200 OK` 
</pre><p>응답 메시지 : HTTP 버전 / 상태 코드</p><p><br></p><p><strong style="color: var(--bold-color);">요청 메서드</strong></p><p>모든 HTTP 요청 메시지는 하나의 메서드를 갖는다. HTTP 요청 메서드는 클라이언트가 서버에 어떤 종류의 동작을 요청하는지를 나타내는 데 사용된다.</p><p><br></p><ul><li><span style="color: var(--bold-color);">GET</span>: 특정 리소스를 조회하기 위한 요청</li><li><span style="color: var(--bold-color);">POST</span>: 서버에 리소스를 생성하도록 요청</li><li><span style="color: var(--bold-color);">PUT</span>: 지정된 URI에 리소스를 저장하도록 서버에 요청</li><li><span style="color: var(--bold-color);">DELETE</span>: 지정된 URI의 리소스를 삭제하도록 요청</li><li><span style="color: var(--bold-color);">HEAD</span>: GET과 유사하지만, 응답 본문(body) 없이 헤더(header) 정보만을 요청. 리소스의 메타데이터를 조회할 때 사용.</li></ul><p><br></p><p><strong style="color: var(--bold-color);">응답 상태 코드</strong></p><p>모든 HTTP 응답 메시지는 상태 코드와 함께 반환된다. 서버가 클라이언트의 요청을 어떻게 처리했는지를 나타내는 숫자 코드이다</p><p><br></p><ul><li><span style="color: var(--bold-color);">200 OK</span>: 요청이 성공적으로 처리되었음을 나타냠. 이는 가장 일반적인 성공 응답 코드.</li><li><span style="color: var(--bold-color);">302 Found (Temporary Redirect)</span>: 이 상태 코드는 요청한 리소스가 일시적으로 다른 URI로 이동되었음을 나타냄. 클라이언트는 이 새 URI로 리디렉션.</li><li><span style="color: var(--bold-color);">404 Not Found</span>: 서버가 요청한 리소스를 찾을 수 없음. 클라이언트가 잘못된 URI를 요청했거나, 요청한 리소스가 삭제되어 더 이상 사용할 수 없을 때 반환.</li></ul><h4><br></h4><h4>2. 헤더</h4><p><br></p><ul><li>시작 줄 다음 0개 이상의 헤더 필드가 이어진다. (문자열로 구성)</li><li>콜론":"으로 구분된 키, 밸류로 구성된다.</li><li>헤더를 추가하기 위해 한 줄을 더하면 된다.</li><li>헤더는 빈줄로 끝난다.</li></ul><h4><br></h4><h4>3. 본문</h4><p><br></p><ul><li>어떤 종류의 데이터든 들어갈 수 있는 메시지 본문</li><li>요청 본문 : 웹서버로 데이터를 전송</li><li>응답 본문: 클라이언트로 데이터 반환</li><li>텍스트, 임의의 이진 데이터(이미지, 비디오, 오디오, 애플리케이션)를 포함할 수 있다.</li></ul><h3><br></h3><h3>6. TCP 커넥션</h3><blockquote>HTTP 메시지는 TCP(Transmission Control Protocol) 커넥션을 통해 한곳에서 다른 곳으로 옮겨진다.</blockquote><h4><br></h4><h4>1. TCP/IP</h4><p><br></p><p><strong>HTTP</strong></p><ul><li>application 계층 프로토콜</li><li>네트워크 통신의 핵심 세부사항은 신경쓰지 않는다</li><li>이에 관한 것은 인터넷 전송 프로토콜 TCP/IP에 맡긴다</li></ul><p><br></p><p><strong>TCP</strong></p><ul><li>전송 계층</li><li>오류 없는 데이터 전송</li><li>순서에 맞는 전달 (데이터는 언제나 보낸 순서대로 도착)</li><li>조각나지 않는 데이터 스트림 (언제든 어떤 크기로든 보낼 수 있음)</li><li>네트워크 개념 상 IP 위의 계층이다.</li></ul><p><br></p><p><strong>TCP/IP</strong></p><ul><li>인터넷 자체가 TCP/IP에 기초하고 있다</li><li>TCP/IP는 TCP와 IP가 층을 이루는 '패킷 교환 네트워크 프로토콜 집합'</li><li>각 네트워크와 하드웨어 특성을 숨기고, 어떤 종류의 컴퓨터나 네트워크든 서로 통신하게 해준다</li></ul><p><br></p><p>TCP 커넥션이 맺어지면, 클라이언트와 서버 간에 교환되는 메시지가 없어지거나, 손상되거나, 순서가 바뀌지 않는다.</p><h5><br></h5><h5><strong>*네트워크 계층 더 알아보기</strong></h5><p><br></p><p>IP</p><ul><li><span style="color: var(--bold-color);">기능</span>: 네트워크 계층에서 IP는 데이터 패킷을 송신지에서 목적지까지 전달하는 역할. 각 디바이스는 고유한 IP 주소를 가지며, 이 주소를 사용하여 데이터를 정확한 목적지로 라우팅.</li><li><span style="color: var(--bold-color);">목적</span>: IP는 전 세계적으로 연결된 다양한 네트워크 상에서 데이터를 안전하고 효율적으로 전송하기 위한 표준 수단을 제공. 패킷 스위칭 기술을 사용하여 정보를 패킷 형태로 분할하고, 이를 네트워크를 통해 전송.</li></ul><p><br></p><p><strong>링크 인터페이스</strong></p><ul><li><span style="color: var(--bold-color);">기능</span>: 링크 인터페이스는 특정 네트워크 매체(예: 이더넷, Wi-Fi)를 통해 데이터를 전송하는 물리적 또는 논리적 장치이다. 네트워크 어댑터나 네트워크 인터페이스 카드(NIC)가 이에 해당.</li><li><span style="color: var(--bold-color);">목적</span>: 링크 인터페이스는 다양한 네트워크 기술과 프로토콜을 사용하여, 디바이스가 로컬 네트워크에 연결되고, 데이터를 교환할 수 있게함. 각각의 인터페이스는 해당 네트워크 매체의 특성에 맞는 신호 변환, 속도 조정, 에러 처리 등의 기능을 수행.</li></ul><p><br></p><p><strong>네트워크 하드웨어</strong></p><p><br></p><ul><li><span style="color: var(--bold-color);">기능</span>: 물리적 네트워크 하드웨어는 네트워크 연결과 데이터 전송을 가능하게 하는 실제 장비를 말한다. 여기에는 라우터, 스위치, 허브, 모뎀 등이 포함.</li><li><span style="color: var(--bold-color);">목적</span>: 네트워크 상에서 데이터를 올바른 목적지로 라우팅하고, 네트워크의 다른 부분과 연결성을 제공. 물리적 하드웨어는 데이터를 전기적, 광학적 또는 무선 신호로 변환하고, 이러한 신호를 네트워크를 통해 전송하는 역할.</li></ul><h4><br></h4><h4>2. 접속, IP 주소, 포트 번호</h4><p><br></p><p>HTTP 클라이언트가 서버에 메시지를 전송하기 전에&nbsp;<span style="color: var(--bold-color);">IP 주소와 포트번호</span>를 사용하여&nbsp;<span style="color: var(--bold-color);">TCP/IP 커넥션</span>을 맺어야한다.</p><p><br></p><ul><li>IP: 호스트 명 (도메인 이름)에서 추출한다. 보통 DNS를 통해 쉽게 IP로 변환된다.</li><li>포트 번호: 보통 생략되는 경우가 많다. http-80, https-443이 기본값</li></ul><p><br></p><p><strong>과정</strong></p><ol><li>웹 브라우저는 서버의 URL에서 호스트 명을 추출</li><li>웹 브라우저는 서버의 호스트 명을 IP로 변환</li><li>웹 브라우저는 URL에서 포트번호를 추출</li><li>IP, 포트번호를 통해 브라우저가 TCP 커넥션을 맺음</li><li>브라우저는 서버에 HTTP 요청을 보냄</li><li>서버는 브라우저에 HTTP 응답을 돌려줌</li><li>커넥션이 닫힘</li><li>브라우저는 문서를 보여줌</li></ol><h3><br></h3><p><br></p><h3>7. 웹 구성요소</h3><p>각 요소는 추후 챕터에서 더 자세히 다룰 예정이다</p><h4><br></h4><h4>1. 프록시</h4><blockquote>클라이언트와 서버 사이에 위치한 HTTP 중계자</blockquote><p><br></p><ul><li>웹 보안, 애플리케이션 통합, 성능 최적화를 위해 사용된다</li><li>클라이언트와 서버 사이에 위치하여 모든 HTTP 요청을 받아 (요청을 수정한 뒤에) 서버에 전달함</li><li>사용자를 위한 중재자로 동작하며 사용자를 대신하여 서버에 접근</li><li>주로 보안을 위해 사용</li><li>요청과 응답을 필터링할 수 있다</li></ul><h3><br></h3><h3>2. 캐시</h3><blockquote>많이 찾는 웹 페이지를 클라이언트 가까이 보관하는 HTTP 창고</blockquote><p><br></p><ul><li>웹 캐시와 캐시 프록시는 자신을 거쳐가는 문서 중 자주 찾는 것의 사본을 저장한다.</li><li>서버보다 클라이언트에 더 가까이 위치한다.</li></ul><p><br></p><p><strong>HTTP 캐시의 장점</strong></p><p><br></p><ul><li>캐시를 효율적으로 저장</li><li>캐시된 컨텐츠를 최신 버전으로 유지</li><li>프라이버시를 보호하기 위한 많은 기능을 정의</li></ul><p><br></p><p>추가 내용: 캐시와 관련된 용어가 많이 사용되어 단어들을 찾아봤다.</p><p><br></p><ul><li><span style="color: var(--bold-color);">HTTP 캐시</span>: HTTP 프로토콜을 사용하여 웹 콘텐츠를 캐시하는 모든 종류의 캐시를 포함하는 광범위한 용어</li><li><span style="color: var(--bold-color);">웹 캐시(Web Cache)</span>: 일반적으로 웹 문서(HTML 페이지, 이미지 등)의 사본을 저장하는 메커니즘을 가리키며, 이는 브라우저 혹은 서버 측 캐시를 포함한다.</li><li><span style="color: var(--bold-color);">캐시 프록시(Cache Proxy)</span>: 캐싱을 주 목적으로 하는 HTTP 프록시 서버의 한 종류</li><li><span style="color: var(--bold-color);">브라우저 캐시(Browser Cache)</span>: 웹 브라우저에 의해 클라이언트 측에서 사용되는 캐시로, 사용자가 방문한 웹 페이지의 사본을 로컬에 저장.</li></ul><h3><br></h3><h3>3. 게이트웨이</h3><blockquote>다른 애플리케이션과 연결된 특별한 웹 서버</blockquote><p><br></p><ul><li><span style="color: var(--bold-color);">다른 서버들의 중개자</span>로 동작하는 특별한 서버</li><li>주로&nbsp;<span style="color: var(--bold-color);">HTTP 트래픽을 다른 프로토콜로 변환</span>하기 위해 사용</li><li>언제나 스스로가 리소스를 갖은 진짜 서버처럼 요청을 다룬다.</li><li>클라이언트에서 게이트웨이와 웹서버를 분간하지 못한다</li><li>ex) HTTP/FTP 게이트웨이는 FTP URI에 대한 HTTP 요청을 받아서 FTP 프로토콜을 사용해 리소스를 가져온다. 이 데이터는 클라이언트에 HTTP 메시지로 전달된다.</li></ul><h3><br></h3><h3>4. 터널</h3><blockquote>단순히 HTTP 통신을 전달하는 특별한 프록시</blockquote><p><br></p><ul><li>두 커넥션 background-color: var(--code-background);">?</span>로 시작하며, <span style="color: var(--code-normal); background-color: var(--code-background);">키=값</span> 쌍의 형태로 정보를 전달</li><li><span style="color: var(--bold-color);">프래그먼트(fragment)</span>: 리소스 내의 특정 부분을 가리키는 선택적인 구성 요소. <span style="color: var(--code-normal); background-color: var(--code-background);">#</span>로 시작.</li></ul><p><br></p><p>URN</p><p>리소스의 위치와 무관하게 리소스를 고유하게 식별하는 URI(Uniform Resource Identifier)의 한 형태. 리소스 위치 대신, 이름을 기반으로 그 자체의 고유성 제공. 예를 들어 RFC(Request for Comments), ISBN(International Standard Book Number) 등이 있다. ex)<em> urn:ietf:rfc:2141</em></p><h3 id="-3"><br></h3><p><br></p><h3 id="4">4. 트랜잭션</h3><blockquote>클라이언트가 웹 서버와 리소스를 주고 받기 위해 HTTP를 사용하는 법</blockquote><p><br></p><p><strong style="color: var(--bold-color);">HTTP 트랜잭션</strong></p><p><br></p><ul><li>요청 명령과 응답 결과로 구성</li><li>요청/응답의 상호작용은 HTTP 메시지라는 데이터 덩어리를 통해 이루어짐</li><li>요청 메시지, 응답 메시지를 통해 요청/응답의 트랜젝션</li><li>ex) 한 개의 웹페이지는 여러 HTTP 트랜젝션을 수행한다</li><li class="ql-indent-1">html을 요청하고 응답을 받는다</li><li class="ql-indent-1">html에서 link, script 등의 태그를 통해 이미지, 스타일, js 스크립트 등 여러 리소스를 요청한다.</li><li class="ql-indent-1">대부분의 경우 하나의 웹페이지는 하나의 리소스가 아닌 리소스의 모음이다.</li></ul><h3 id="-4"><br></h3><p><br></p><h3 id="5-http">5. HTTP 메시지</h3><p><br></p><p>각 소항목의 내용들은 Chapter 3에서 더 자세히 다룰 예정이다. 개괄적으로 보면 다음과 같다.</p><p><br></p><ul><li>HTTP 메시지는 단순한 줄 단위의 문자열이다.</li><li>요청, 응답 메시지 이외에 HTTP 메시지는 없다.</li><li>요청 메시지: 클라이언트에서 서버로 보내는 메시지</li><li>응답 메시지: 서버에서 클라이언트로 보내는 메시지</li></ul><p><br></p><p>HTTP 메시지는 3가지로 구성된다.</p><h4><br></h4><h4>1. 시작줄</h4><p>메시지의 첫 줄에 해당하며, 요청/응답은 다음과 같이 구성된다.</p><p><br></p><pre class="ql-syntax pre" spellcheck="false"><code class="hljs language-bash">`GET /index.html HTTP/1.1` 
</code></pre><p>요청 메시지 : HTTP 메서드 / 요청 대상(URI) / HTTP 버전</p><pre class="ql-syntax pre" spellcheck="false"><code class="hljs language-markdown"><span class="hljs-bullet">-</span> <span class="hljs-code">`HTTP/1.1 200 OK`</span> 
</code></pre><p>응답 메시지 : HTTP 버전 / 상태 코드</p><p><br></p><p><strong style="color: var(--bold-color);">요청 메서드</strong></p><p>모든 HTTP 요청 메시지는 하나의 메서드를 갖는다. HTTP 요청 메서드는 클라이언트가 서버에 어떤 종류의 동작을 요청하는지를 나타내는 데 사용된다.</p><p><br></p><ul><li><span style="color: var(--bold-color);">GET</span>: 특정 리소스를 조회하기 위한 요청</li><li><span style="color: var(--bold-color);">POST</span>: 서버에 리소스를 생성하도록 요청</li><li><span style="color: var(--bold-color);">PUT</span>: 지정된 URI에 리소스를 저장하도록 서버에 요청</li><li><span style="color: var(--bold-color);">DELETE</span>: 지정된 URI의 리소스를 삭제하도록 요청</li><li><span style="color: var(--bold-color);">HEAD</span>: GET과 유사하지만, 응답 본문(body) 없이 헤더(header) 정보만을 요청. 리소스의 메타데이터를 조회할 때 사용.</li></ul><p><br></p><p><strong style="color: var(--bold-color);">응답 상태 코드</strong></p><p>모든 HTTP 응답 메시지는 상태 코드와 함께 반환된다. 서버가 클라이언트의 요청을 어떻게 처리했는지를 나타내는 숫자 코드이다</p><p><br></p><ul><li><span style="color: var(--bold-color);">200 OK</span>: 요청이 성공적으로 처리되었음을 나타냠. 이는 가장 일반적인 성공 응답 코드.</li><li><span style="color: var(--bold-color);">302 Found (Temporary Redirect)</span>: 이 상태 코드는 요청한 리소스가 일시적으로 다른 URI로 이동되었음을 나타냄. 클라이언트는 이 새 URI로 리디렉션.</li><li><span style="color: var(--bold-color);">404 Not Found</span>: 서버가 요청한 리소스를 찾을 수 없음. 클라이언트가 잘못된 URI를 요청했거나, 요청한 리소스가 삭제되어 더 이상 사용할 수 없을 때 반환.</li></ul><h4><br></h4><h4>2. 헤더</h4><p><br></p><ul><li>시작 줄 다음 0개 이상의 헤더 필드가 이어진다. (문자열로 구성)</li><li>콜론":"으로 구분된 키, 밸류로 구성된다.</li><li>헤더를 추가하기 위해 한 줄을 더하면 된다.</li><li>헤더는 빈줄로 끝난다.</li></ul><h4><br></h4><h4>3. 본문</h4><p><br></p><ul><li>어떤 종류의 데이터든 들어갈 수 있는 메시지 본문</li><li>요청 본문 : 웹서버로 데이터를 전송</li><li>응답 본문: 클라이언트로 데이터 반환</li><li>텍스트, 임의의 이진 데이터(이미지, 비디오, 오디오, 애플리케이션)를 포함할 수 있다.</li></ul><h3 id="-5"><br></h3><h3 id="6-tcp">6. TCP 커넥션</h3><blockquote>HTTP 메시지는 TCP(Transmission Control Protocol) 커넥션을 통해 한곳에서 다른 곳으로 옮겨진다.</blockquote><h4><br></h4><h4>1. TCP/IP</h4><p><br></p><p><strong>HTTP</strong></p><ul><li>application 계층 프로토콜</li><li>네트워크 통신의 핵심 세부사항은 신경쓰지 않는다</li><li>이에 관한 것은 인터넷 전송 프로토콜 TCP/IP에 맡긴다</li></ul><p><br></p><p><strong>TCP</strong></p><ul><li>전송 계층</li><li>오류 없는 데이터 전송</li><li>순서에 맞는 전달 (데이터는 언제나 보낸 순서대로 도착)</li><li>조각나지 않는 데이터 스트림 (언제든 어떤 크기로든 보낼 수 있음)</li><li>네트워크 개념 상 IP 위의 계층이다.</li></ul><p><br></p><p><strong>TCP/IP</strong></p><ul><li>인터넷 자체가 TCP/IP에 기초하고 있다</li><li>TCP/IP는 TCP와 IP가 층을 이루는 '패킷 교환 네트워크 프로토콜 집합'</li><li>각 네트워크와 하드웨어 특성을 숨기고, 어떤 종류의 컴퓨터나 네트워크든 서로 통신하게 해준다</li></ul><p><br></p><p>TCP 커넥션이 맺어지면, 클라이언트와 서버 간에 교환되는 메시지가 없어지거나, 손상되거나, 순서가 바뀌지 않는다.</p><h5><br></h5><h5><strong>*네트워크 계층 더 알아보기</strong></h5><p><br></p><p>IP</p><ul><li><span style="color: var(--bold-color);">기능</span>: 네트워크 계층에서 IP는 데이터 패킷을 송신지에서 목적지까지 전달하는 역할. 각 디바이스는 고유한 IP 주소를 가지며, 이 주소를 사용하여 데이터를 정확한 목적지로 라우팅.</li><li><span style="color: var(--bold-color);">목적</span>: IP는 전 세계적으로 연결된 다양한 네트워크 상에서 데이터를 안전하고 효율적으로 전송하기 위한 표준 수단을 제공. 패킷 스위칭 기술을 사용하여 정보를 패킷 형태로 분할하고, 이를 네트워크를 통해 전송.</li></ul><p><br></p><p><strong>링크 인터페이스</strong></p><ul><li><span style="color: var(--bold-color);">기능</span>: 링크 인터페이스는 특정 네트워크 매체(예: 이더넷, Wi-Fi)를 통해 데이터를 전송하는 물리적 또는 논리적 장치이다. 네트워크 어댑터나 네트워크 인터페이스 카드(NIC)가 이에 해당.</li><li><span style="color: var(--bold-color);">목적</span>: 링크 인터페이스는 다양한 네트워크 기술과 프로토콜을 사용하여, 디바이스가 로컬 네트워크에 연결되고, 데이터를 교환할 수 있게함. 각각의 인터페이스는 해당 네트워크 매체의 특성에 맞는 신호 변환, 속도 조정, 에러 처리 등의 기능을 수행.</li></ul><p><br></p><p><strong>네트워크 하드웨어</strong></p><p><br></p><ul><li><span style="color: var(--bold-color);">기능</span>: 물리적 네트워크 하드웨어는 네트워크 연결과 데이터 전송을 가능하게 하는 실제 장비를 말한다. 여기에는 라우터, 스위치, 허브, 모뎀 등이 포함.</li><li><span style="color: var(--bold-color);">목적</span>: 네트워크 상에서 데이터를 올바른 목적지로 라우팅하고, 네트워크의 다른 부분과 연결성을 제공. 물리적 하드웨어는 데이터를 전기적, 광학적 또는 무선 신호로 변환하고, 이러한 신호를 네트워크를 통해 전송하는 역할.</li></ul><h4><br></h4><h4>2. 접속, IP 주소, 포트 번호</h4><p><br></p><p>HTTP 클라이언트가 서버에 메시지를 전송하기 전에 <span style="color: var(--bold-color);">IP 주소와 포트번호</span>를 사용하여 <span style="color: var(--bold-color);">TCP/IP 커넥션</span>을 맺어야한다.</p><p><br></p><ul><li>IP: 호스트 명 (도메인 이름)에서 추출한다. 보통 DNS를 통해 쉽게 IP로 변환된다.</li><li>포트 번호: 보통 생략되는 경우가 많다. http-80, https-443이 기본값</li></ul><p><br></p><p><strong>과정</strong></p><ol><li>웹 브라우저는 서버의 URL에서 호스트 명을 추출</li><li>웹 브라우저는 서버의 호스트 명을 IP로 변환</li><li>웹 브라우저는 URL에서 포트번호를 추출</li><li>IP, 포트번호를 통해 브라우저가 TCP 커넥션을 맺음</li><li>브라우저는 서버에 HTTP 요청을 보냄</li><li>서버는 브라우저에 HTTP 응답을 돌려줌</li><li>커넥션이 닫힘</li><li>브라우저는 문서를 보여줌</li></ol><h3 id="-6"><br></h3><p><br></p><h3 id="7">7. 웹 구성요소</h3><p>각 요소는 추후 챕터에서 더 자세히 다룰 예정이다</p><h4><br></h4><h4>1. 프록시</h4><blockquote>클라이언트와 서버 사이에 위치한 HTTP 중계자</blockquote><p><br></p><ul><li>웹 보안, 애플리케이션 통합, 성능 최적화를 위해 사용된다</li><li>클라이언트와 서버 사이에 위치하여 모든 HTTP 요청을 받아 (요청을 수정한 뒤에) 서버에 전달함</li><li>사용자를 위한 중재자로 동작하며 사용자를 대신하여 서버에 접근</li><li>주로 보안을 위해 사용</li><li>요청과 응답을 필터링할 수 있다</li></ul><h3 id="-7"><br></h3><h3 id="2-1">2. 캐시</h3><blockquote>많이 찾는 웹 페이지를 클라이언트 가까이 보관하는 HTTP 창고</blockquote><p><br></p><ul><li>웹 캐시와 캐시 프록시는 자신을 거쳐가는 문서 중 자주 찾는 것의 사본을 저장한다.</li><li>서버보다 클라이언트에 더 가까이 위치한다.</li></ul><p><br></p><p><strong>HTTP 캐시의 장점</strong></p><p><br></p><ul><li>캐시를 효율적으로 저장</li><li>캐시된 컨텐츠를 최신 버전으로 유지</li><li>프라이버시를 보호하기 위한 많은 기능을 정의</li></ul><p><br></p><p>추가 내용: 캐시와 관련된 용어가 많이 사용되어 단어들을 찾아봤다.</p><p><br></p><ul><li><span style="color: var(--bold-color);">HTTP 캐시</span>: HTTP 프로토콜을 사용하여 웹 콘텐츠를 캐시하는 모든 종류의 캐시를 포함하는 광범위한 용어</li><li><span style="color: var(--bold-color);">웹 캐시(Web Cache)</span>: 일반적으로 웹 문서(HTML 페이지, 이미지 등)의 사본을 저장하는 메커니즘을 가리키며, 이는 브라우저 혹은 서버 측 캐시를 포함한다.</li><li><span style="color: var(--bold-color);">캐시 프록시(Cache Proxy)</span>: 캐싱을 주 목적으로 하는 HTTP 프록시 서버의 한 종류</li><li><span style="color: var(--bold-color);">브라우저 캐시(Browser Cache)</span>: 웹 브라우저에 의해 클라이언트 측에서 사용되는 캐시로, 사용자가 방문한 웹 페이지의 사본을 로컬에 저장.</li></ul><h3 id="-8"><br></h3><h3 id="3-1">3. 게이트웨이</h3><blockquote>다른 애플리케이션과 연결된 특별한 웹 서버</blockquote><p><br></p><ul><li><span style="color: var(--bold-color);">다른 서버들의 중개자</span>로 동작하는 특별한 서버</li><li>주로 <span style="color: var(--bold-color);">HTTP 트래픽을 다른 프로토콜로 변환</span>하기 위해 사용</li><li>언제나 스스로가 리소스를 갖은 진짜 서버처럼 요청을 다룬다.</li><li>클라이언트에서 게이트웨이와 웹서버를 분간하지 못한다</li><li>ex) HTTP/FTP 게이트웨이는 FTP URI에 대한 HTTP 요청을 받아서 FTP 프로토콜을 사용해 리소스를 가져온다. 이 데이터는 클라이언트에 HTTP 메시지로 전달된다.</li></ul><h3 id="-9"><br></h3><h3 id="4-1">4. 터널</h3><blockquote>단순히 HTTP 통신을 전달하는 특별한 방화벽.
